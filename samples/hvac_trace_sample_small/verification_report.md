## 1. 検証方法

### 1.1 実行環境の制約と対処

3ツールはいずれも実行時にpdf.js／xlsx.js／tiny-segmenter／cytoscape.jsをCDN（cdnjs.cloudflare.com、unpkg.com、cdn.jsdelivr.net）から動的読込する設計である。検証セッションのネットワークポリシーでは、これら汎用CDNへの直接アクセスが拒否された（`unpkg.com`等への接続が403）。

対処として、npmレジストリ（アクセス許可対象）から**ツール指定と同一バージョン**のライブラリを取得し、Playwright（Chromium）のルートインターセプトでCDN URLをローカルファイルに差し替えて実行した。

| ライブラリ | ツール指定バージョン | 取得元 |
|---|---|---|
| pdfjs-dist | 3.11.174 | npm registry |
| xlsx (SheetJS) | 0.18.5 | npm registry |
| tiny-segmenter | 0.2.0 | npm registry |
| cytoscape | 3.26.0 | npm registry |

**留意点:** この方式はライブラリのバージョン・内容をCDN配布物と一致させているが、実際の本番ブラウザ環境（CDNへ到達可能）での動作を100%代替するものではない。CDNの可用性・レスポンス遅延・改ざん検知などはこの検証の対象外。

### 1.2 操作方法

Playwrightで各ツールのHTMLファイルを`file://`で開き、実際のUI要素（ファイル選択、ボタン、入力欄）を操作した。一部、大量の行を人手で1件ずつクリックする代わりに、ツール自身が内部で使っている状態更新関数（`v12ReviewEntry`等、UIのonchangeハンドラが呼ぶのと同じ関数）を`page.evaluate`経由で複数件まとめて呼び出した箇所がある（レビュー状態・タグの一括設定）。該当箇所は本文中に明記する。

最終盤の検証（4章）では、これらの状態操作を一切使わず、**「様式ファイル登録」ボタンによる正規のファイルインポート経路のみ**で再現できることを別途確認している。

## 3. 根本原因の切り分け：ツール生成物とサンプルJSONの差異

「2.3節で使ったサンプルJSONは、PDF/Excelをツールへそのまま流し込んだ生成物と同じか」を確認する目的で、既定様式のままPDF/Excelから照合用JSONを再生成し、付属サンプルJSONと突き合わせた。

### 3.1 結果：一致しない

| 項目 | サンプルJSON | 既定様式での生成物 |
|---|---|---|
| PDF側レコード数 | 4件（要求のみ） | 15件（書誌情報・章見出し・注記・ページ番号を含む生の抽出全部） |
| PDF側`tags` | 付与済み（例: `["使用温度"]`） | 空配列（タグ付け作業をしていないため） |
| PDF側`review_status` | `reviewed` | `unreviewed` |
| Excel側`trace_title` | 設計項目列の値（例:`"使用温度範囲"`） | 分類列の値（例:`"環境"`）——既定様式では列選択が異なるため |
| Excel側`trace_text` | 4列のみ連結 | 8列全部を連結 |
| Excel側`tags` | 付与済み | 空配列 |

本文（`trace_text`が参照する元セル値＝`source_record`）自体は完全一致しており、ツールの読み取り・抽出そのものは正確である。差異はすべて「様式プロファイルの列選択」と「タグ辞書の未登録」に起因する。

### 3.2 対処：様式プロファイル＋タグ辞書を用意して再検証

`usage_guide.md`推奨の4列（設計項目／標準機種情報／顧客対応・検討内容／検討結果）を選ぶExcel用様式プロファイルと、共通タグ辞書（`tag_policy.allowed_tags`＋Excel側は`rules`）を作成し、両ツールへ登録した上で以下を実施。

- Excel: 様式適用→「照合タグ自動付与」→「未確認を一括確認」→「照合用JSON出力」
- PDF: 章見出し・書誌情報・注記・ページ番号など11ブロックを「除外」レビュー、要求4件にタグ付与・確認済み化→「照合用JSON出力」

**結果: PDF4件・Excel5件とも、`trace_text`が一字一句完全一致。タグも完全一致（Excel側1件のみ配列の順序違い、内容は同一）。**

差異が残ったのは`trace_id`のみ（サンプルは`req-use-temperature`等の人間可読な命名、ツール生成物は`blk-xxx`/`excel-xxx`のハッシュ自動採番）。後段のA/B照合エンジンは`trace_id`ではなく`trace_text`・タグ・キーワードで照合するため、この差は照合結果に影響しない。

### 3.3 課題の切り分け結論

| 論点 | 結論 |
|---|---|
| ツールの機能改修が必要か | **不要**。既存機能のみで両ツールとも完全再現できた |
| タグ辞書ファイルが必要か | **必要**。事前に共通タグ辞書（`allowed_tags`）を登録しないと、付与したタグが全て`unregistered_tags`に落ちる（検証中に実際に再現した） |
| 様式プロファイルが必要か | Excel側は**必要**（8列中どの4列を照合本文にするか指定するため）。PDF側は既定プロファイルの流用で足り、**タグ辞書の追記だけで十分** |
| そもそもツール改修が必要か | **不要**。差異は運用時の設定・レビュー作業の省略が原因で、バグや機能不足ではなかった |

### 3.4 事前準備ファイルの正規インポート経路での再確認

3.2節の検証は一部、`page.evaluate`でツール内部の状態を直接操作した箇所を含む（大量ブロックのレビュー状態一括設定）。これが「UI操作の代替として妥当か」を確認するため、実際に**「様式ファイル登録」ボタンでのファイルインポート**のみを使う経路で再検証した。

- `samples/hvac_trace_sample_small/profile_pdf_customer_requirements.json`をPDFツールの「様式ファイル登録」から読込
- `samples/hvac_trace_sample_small/profile_excel_design_review.json`をExcelツールの「様式ファイル登録」から読込
- 上記以外の操作（変換、タグ自動付与／個別タグ付与、除外レビュー、照合用JSON出力）は通常のUI操作

**結果: 両ツールとも、様式ファイルをインポートしただけで`unregistered_tags`が0件になり、警告ダイアログなしで出力に成功。出力内容はPDF4件・Excel5件とも`trace_text`・タグが完全一致した。**

これにより、「様式プロファイル＋タグ辞書のファイルを事前に配布しておけば、他の利用者もインポートするだけで同じ結果を再現できる」ことを実運用に近い形で確認した。

## 4. 残課題・注意点（ツール改修は不要だが、認識しておくべき事項）

いずれも今回の検証で「バグ」として再現・確定させたものではなく、コード上の設計として確認できた既知の制約、または実利用時に注意すべき挙動である。

### 4.1 `trace_id`の命名方式（軽微・機能への影響なし）

ツールは`trace_id`を常にハッシュベースで自動採番する（PDF: `blk-xxx`、Excel: `excel-xxx`）。人間可読なID（`req-use-temperature`等）を生成する機能は無い。A/B照合エンジンは`trace_id`を照合キーに使わないため、照合結果には影響しない。表示・追跡のしやすさのみに関わる制約。

### 4.2 PDF/Excel間で様式プロファイルを1ファイル化できない

PDF側プロファイルは`display`/`extraction`構造、Excel側プロファイルは`output`/`columns`構造で、トップレベルが非互換（各ツールの様式仕様書にも明記）。共通タグ辞書を使う場合、`tag_policy`部分の値（`vocabulary_id`・`allowed_tags`）を両ファイルで手動同期する必要がある。タグを追加・変更する際、片方の更新漏れが起きうる運用上のリスクとして認識しておくこと。

### 4.3 PDF側にキーワード自動タグ付与機能が無い（ツール間の非対称性）

Excel側の`tag_policy.rules`（キーワード一致でタグを自動付与）に相当する仕組みが、PDF側の`tag_policy`には存在しない（コード確認済み。PDF側`tag_policy`は`allowed_tags`/`tag_aliases`/`suggested_tags`のみ）。PDF側のタグ付けは人手、または様式編集画面のAI候補提示（頻度ベースのヒューリスティック、辞書ルールではない）に依存する。案件で要求文章数が多い場合、PDF側のタグ付け工数がExcel側より大きくなる可能性がある。

### 4.4 自動照合の分類ロジック：`tag`等の方式は常に「要確認」

`json_ab_trace_matching_tool_v12.1.15.html`の`traceJudgement(method, confidence)`関数を確認したところ、`method`が`exact`または`code`の場合のみ「対応あり」と判定し、それ以外（`tag`／`vector`／`synonym`／`fuzzy`等、信頼度の値に関わらず）は一律「要確認」に分類される設計だった。

```js
function traceJudgement(method, confidence) {
  if (method === 'exact' || method === 'code') return '対応あり';
  if (!method || method === 'none') return '未対応';
  return '要確認';
}
```

2章で確認した信頼度0.88のタグ一致4件が全て「要確認」に分類されたのはこの仕様による（属性ゲート等の副作用ではない）。意図的な保守的設計（意味的な一致は常に人間確認を要求する）と考えられ、`usage_guide.md`の運用方針とも整合するが、**この閾値・分類基準がツールのHELP等で利用者に明示されているか**は別途確認が望ましい。

### 4.5 PDF/Excelツールで「未確認のまま出力」の扱いが非対称

- PDFツール: 未確認項目が残っていても、確認ダイアログ（「作業用として出力しますか？」）で承諾すれば出力できる
- Excelツール: 未確認行が1件でも残っていると、確認ダイアログなしでエラー表示となり出力を**強制的にブロック**する（抜け道なし）

意図的な設計差の可能性があるが、統一されていない点として記録しておく。運用上は問題にならない（最終的に両ツールとも「未確認のまま出力しない」運用を徹底すればよい）が、ツールの仕様書またはHELPにこの非対称性の記載があるか確認するとよい。

### 4.6 Excel側入力欄の自動連動によるテスト時の入力ミス（ツールのバグではないと判断）

検証中、「データ開始行」欄へ値を入力した際に意図せず「55」という値になった事象があった。コードを確認したところ、「ヘッダー行」欄の`change`イベントで、現在の「データ開始行」の値がヘッダー行以下であれば自動的に「ヘッダー行+1」へ更新する仕様だった。

```js
headerRow.addEventListener('change', () => {
  ...
  if (Number(dataStartRow.value) <= value) dataStartRow.value = value + 1;
```

検証スクリプトがこの自動更新後に既存値を選択せずキー入力を追加したために生じた事象であり、**手動UI操作でこの不具合が再現するかは未確認**。実利用者が数値欄を編集する際は、既存値を選択してから入力する（もしくは値をクリアしてから入力する）ことを推奨する運用上の注意点として記録する。

### 4.7 Excel側`照合用JSON`のスキーマがサンプルJSONより情報量が多い

ツールが実際に出力する`excel-row-trace-v1`（schema 1.1）には、`stable_uid`／`content_hash`／`review_method`／`reviewed_at`／`review_comment`／`exclusion_reason`が含まれるが、サンプル`JSON_B_design_review_trace.json`にはこれらが無い。サンプルはREADME内で「模擬した」データと明記されている簡略版であり、ツールの実出力とは元々スキーマの充実度が異なることを利用者が誤解しないよう、`usage_guide.md`側で明記済み。

## 6. レビュー観点の提案

他AI／人間レビュアーへの依頼事項として、特に次の点を重点確認いただきたい。

1. 4.4節の分類ロジック（`tag`等の方式が信頼度によらず常に「要確認」になる仕様）が、ツールのHELPや利用者向けドキュメントに明記されているか。明記がなければ、`usage_guide.md`だけでなくツール本体のHELPにも追記が必要か。
2. 4.5節の未確認出力ブロックの非対称性（PDF側は承諾で回避可、Excel側は完全ブロック）が意図的な仕様か、統一すべき差異か。
3. 4.2節の「タグ辞書をプロファイル2ファイルへ手動同期する」運用が、タグ辞書変更時の更新漏れリスクとして許容範囲か。将来的に共有タグ辞書ファイルを読み込む機能（`$ref`的な仕組み）を両ツールへ追加すべきか。
4. 1.1節のCDN代替検証方法が、本番相当のネットワーク環境での動作を十分に代替できているか（別途、CDNへ到達可能な環境での確認を推奨）。
