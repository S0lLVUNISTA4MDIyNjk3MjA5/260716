## 1. なぜこの順序で着手したか

`abstraction_levels.md`のレビューでは、実装優先順位として「意味対応付け（工程2/3）を数量構造化（工程4）より先に」という提案があった。一方で、現行ツール群の実装状況を見ると、

- 構造抽出（章・節、Excel行列）：**実装済み**、検証済み
- タグ付け・照合エンジン（exact/code/tag/vector/synonym/fuzzy、確信度階層）：**実装済み**、検証済み
- 数量・単位の抽出と比較：**未着手**（現状は`trace_text`という1本の文字列の一部として埋もれており、構造化されたことが一度もない）

という非対称がある。他の工程は「改良ベースで機能アップ」できる状態にあるのに対し、数量抽出だけは概念（`abstraction_levels.md`の工程4）のまま何も検証されていなかった。そこで、**実装ロードマップの優先順位を変更するのではなく**、まず数量抽出を「同じ土俵」に乗せる目的で、実データを使った動くたたき台を作成した。

## 2. スキーマ（v2.1、レビューの推奨スキーマを採用）

初版は`value.kind: "single"|"range"|"alternatives"`という3種類の形を持っていたが、レビュー4章の指摘に従い、単一値・片側制約・範囲を**区間（interval）モデルへ統一**した。単一値は`lower === upper`の退化区間として表現する。

```js
{
  source_text: "５０℃～６０℃",       // 元テキストそのもの(全角等を正規化していない)
  normalized_text: "50℃~60℃",       // 正規化後の文字列(全角→半角、～→~等)
  quantity: {
    kind: "interval" | "alternatives",
    lower: { value: 50, inclusive: true } | null,   // 片側のみの制約(以下/未満)ならnull
    upper: { value: 60, inclusive: true } | null,   // 片側のみの制約(以上/超える)ならnull
    options: [50, 60],               // kind:"alternatives"のとき(例: "50/60 Hz")
    selection_semantics: "unknown",  // alternativesの意味(択一/両対応等)は確定しない
  },
  unit: { source: "℃", canonical: "degC", dimension: "temperature" },
  qualifiers: [{ type: "approximate" | "maximum" | "minimum" | "nominal", source_text: "約" }], // あれば
  context: {
    property: null, subject: null, state: null,   // 工程3(意味対応付け)が埋める領域。本工程では確定しない
    tokens: ["AC", "三相"],           // 周辺語(定格/交流/直流等)をそのまま保持
  },
  condition_candidates: [{ source_text: "50 °C", quantity: {...}, unit: {...}, confidence: 0.7 }], // あれば
  extraction: { confidence: 0.95, warnings: [] },
}
```

`condition_candidates`（複数形・確信度付き）が本たたき台の要点である。「周囲温度50 °Cにおいて、冷房能力12 kW以上」のような複文から、「12kW以上」という値が「周囲温度50°C」という条件付きであることを分離して保持する。条件を無視して数値だけ比較すると、異なる前提条件下の値を誤って比較してしまう（4章参照）。v2では**同一文内に限定**して条件候補を付与し、句点をまたいだ誤伝播を防いでいる（5.1節参照）。

`samples/hvac_trace_sample_small/`の実文14件（PDF要求4件＋Excel標準機種情報/検討結果 各5件）に加え、レビューが提示した境界系・失敗系テストケース21件（5章参照）を対象に抽出した。全件の抽出結果は`quantity_extraction_prototype.js`を実行すれば再現できる。代表例のみ示す（v2のスキーマで表記）。

## 4. 工程5（比較・横断分析）への橋渡し：温度範囲の充足判定

抽出したPDF要求とExcel側の値を実際に比較し、`abstraction_levels.md`4章で示した「関連性・標準適合性・対応後適合性を分離する」の温度例を、コードで再現した。v2では、レビュー2.5の指摘に従い比較結果が**暫定判定であること自体を構造化して明示**する。

```text
PDF要求:        interval(lower=0, upper=50, degC)
Excel標準機種:   interval(lower=0, upper=40, degC)
Excel検討結果:   interval(lower=0, upper=50, degC)

要求 vs 標準機種: { comparable:true, provisional:true, satisfied:false, highGap:10, assumptions:[...] } → 「10℃不足」
要求 vs 検討結果: { comparable:true, provisional:true, satisfied:true,  highGap:0,  assumptions:[...] } → 「充足」
```

`assumptions`には「同じ設計特性として選択済み」「同じ運転条件」「単位換算不要」「抽出結果に要確認事項なし」という、この比較が成立するための前提が明示される。これらの前提が崩れている場合（設計特性が実は違う、運転条件が違う等）、比較結果は無条件に信頼してはならないことをデータ構造自体で示す。

これは、①数量を構造化して取り出し、②単位が一致するか確認し、③区間同士の被覆判定を計算する、という3手順だけで得られた結果であり、AIを一切使用していない。`abstraction_levels.md`で述べた「工程4（抽出）は非AIで可能」という主張を、実データで裏付けた。

**ただし注意点**：この比較は「PDF要求の`range(0,50)`」と「Excel標準機種の`range(0,40)`」が**同じ設計特性を指している**という前提のもとでのみ成立する。この前提の判定自体（工程3・意味対応付け）は、本たたき台の範囲外であり、今回は人間が読んで対応させた2つの文字列を直接渡している。数量抽出（工程4a）は意味対応付け（工程3）の代替にはならず、あくまで工程3が終わった後（または既存のタグ一致候補に対する補強シグナルとして）機能するものである。

## 5. 発見した不具合・限界

### 5.1 初版で発見・修正済みの不具合

初版たたき台を実際に動かしたことで、設計段階では気づかなかった問題が見つかった。「概念だけでは分からない、実装して初めて分かること」の実例として記録する。

**範囲表現で単位が両側にある形式を取りこぼしていた**：初版の正規表現は`X~Y unit`（単位1回）だけを想定していたが、実データの大半は`0 °C～40 °C`のように**単位が両側にある**表記だった。これは本たたき台のフラグシップ例（温度範囲の比較）そのものであり、修正しなければ4章の比較デモが成立しなかった。

### 5.2 レビューで発見され、v2で修正済みの不具合

`quantity_extraction_prototype_review.md`によるレビューで、初版になお残っていた問題が6件指摘された。全件をコードで再現・検証したうえで修正した（検証方法・結果は同レビュー文書の「対応記録」節を参照）。

| # | 問題 | 修正内容 |
|---|---|---|
| 1 | 符号付き数値（`-10℃`等）を抽出できない | ASCIIハイフン・全角マイナス・Unicodeマイナスを正規化し、符号付き数値に対応 |
| 2 | `0℃以上50℃以下`が2つの独立した単一値になる | 片側制約の隣接ペアを1つの区間へ統合するロジックを追加 |
| 3 | 境界の包含・非包含（以上/以下 vs 未満/超える）を保持していない | `{value, inclusive}`構造へ変更 |
| 4 | `raw_text`が正規化後の文字列で、真の原文ではない | 正規化を1:1文字置換に限定し、`source_text`（原文）と`normalized_text`（正規化後）を分離して保持 |
| 5 | 比較結果（`coverageGap()`）が暫定判定であることを構造化していない | `provisional: true`と`assumptions`配列を明示 |
| 6 | 条件節（`condition`）が文をまたいで誤伝播する（`50℃で停止する。電源は220Vとする`で実際に再現した） | 文単位に分割してから条件候補を付与するよう変更。かつ`condition`→`condition_candidates`（複数・確信度付き）へ変更し、自動確定ではなく候補として保持 |

推奨修正（3.1〜3.4）についても、単位比較のcanonical化、スラッシュ表現の意味を確定しない（`selection_semantics: "unknown"`）、周辺語(`context.tokens`)の保持、逆転範囲の検出を反映した。

### 5.3 レビュー対応中に追加で発見した不具合（レビュー未記載）

v2の実装・テスト過程で、レビューが明示的には挙げていなかった2件の不具合を独自に発見し、修正した。いずれも「`needs_review`フラグすら立たず、黙って誤った数値を返す」という、フラグ付きの不確実性より危険な種類の不具合だった。

- **`50±2℃`のような公差表記で、中心値（50）が失われ公差（2）だけが抽出される**：`±`を考慮しない数値正規表現が「2℃」だけにマッチしていた。修正後は`interval(lower=48, upper=52)`として区間化し、公差由来であることを`warnings`に記録する。
- **`1,500 mm`のような桁区切りカンマで、「1」が失われ「500 mm」になる**：数値正規表現がカンマ区切りを考慮していなかった。修正後はカンマ区切りを許容する数値トークンへ変更した。

### 5.4 v2.1で追加対応した事項

v2に対する再レビューで、スキーマには境界の包含・非包含が存在する一方、`coverageGap()`が数値だけを比較し、境界が等しい場合の`inclusive`を考慮していないことが判明した。また、`約50℃`・`最大50℃`のような修飾語が`source_text`から外れ、`最大50℃`が完全一致値`[50,50]`として扱われていた。

v2.1では次を追加した。

- 要求境界を含むが実仕様境界が含まない場合、被覆不足と判定する。
- 片側が無限区間の場合を明示的に処理する。
- 数値差が0でも境界包含が異なる場合を`boundaryMismatch`で示す。
- `約`・`およそ`・`程度`・`最大`・`最小`・`公称`を`qualifiers`として保持する。
- 修飾語を`source_text`／`normalized_text`の抽出範囲へ含める。
- `最大`は上限候補、`最小`は下限候補として抽出し、文脈による確定が必要であることを警告する。
- 既存18件に、修飾語3件と境界被覆2件を追加し、合計23件の自動アサーションが全件成功することを確認した。

### 5.5 v2.2で追加対応した事項

v2.1では、`最大`/`最小`由来の片側境界（`_boundSide`）が、`以上`/`以下`由来の片側境界と異なり隣接統合の対象外になっていた。「0℃以上50℃以下」は1件の区間へ統合される一方、「最小0℃最大50℃」は確信度0.6の片側候補2件のまま残る、という非対称な挙動になっており、これが意図的な安全設計（曖昧な修飾語同士は安易に統合しない）か、単純な実装漏れかがコード・文書のどちらにも記載されていなかった。

検討の結果、`以上`/`以下`と同様に統合する方針とした。`_boundSide`を`最大`/`最小`由来の場合にも設定するよう修正し、隣接する片側境界を区間へ統合するようにした。統合後も、各片側候補が持っていた`「最大」を片側境界候補として抽出。文脈による確定が必要`等の`warnings`はそのまま引き継がれるため、**統合後の区間も確信度は自動的に低いまま（0.6）保たれる**（区間として扱えるようにはなるが、無条件に信頼してよい値にはならない）。

既存23件に、「最小0℃最大50℃」が1件の区間に統合されること、統合後も確信度が0.6のままであることを検証する2件を追加し、合計25件の自動アサーションが全件成功することを確認した。単独で現れる`最大50℃`・`最小10kW`（隣接する反対側の境界がない場合）が誤って無関係な他の数量と結合されないことも回帰確認済み。

### 5.6 v2.3で発見・修正した重大な不具合（工程3プロトタイプの実データ検証で発見）

工程3（意味対応付けの候補生成、`semantic_mapping_prototype.js`）のたたき台を実データで動かした際に、v2.2までレビュー・25件の自動アサーションを通過していた「基準版」に、**サンプルデータの主要な値の一つが黙って破損する**重大な不具合が見つかった。

**症状**：`samples/hvac_trace_sample_small/`の実文「周囲温度50 °Cで12.5 kW」（Excel「冷房能力」行の検討結果）を抽出すると、`12.5 kW`ではなく**`5 kW`**が抽出されていた（`12.`の部分が失われる）。

**原因**：`splitSentences()`が文の区切りとして`.`（ASCIIピリオド）を無条件に扱っており、`12.5`の小数点を文の区切りと誤認識していた。その結果、「周囲温度50 °Cで12.5 kW」が「周囲温度50 °Cで12.」と「5 kW」という2つの別々の「文」に分割され、後半の「5 kW」が単独の値として抽出されていた。この不具合は`needs_review`等のフラグも一切立たず、`extraction.confidence`も通常どおり`0.95`（警告なし）のまま出力される、**最も気づきにくい種類の不具合**だった。

**なぜv2.1／v2.2のレビューで見逃されたか**：この不具合は、文分割（v2.1で条件節の誤伝播防止のために導入）と小数点処理（初版から存在）という、別々に見れば正しく動く2つの機能の**組み合わせ**で初めて発生する。既存の自動アサーション25件は、小数点を含む値と文分割が同時に絡むケースを1件もカバーしていなかった。デモ出力には正しくこの誤った結果（`5 kW`）が表示され続けていたが、アサーションの成否だけを追っていたため見過ごされていた。工程3のたたき台で実データ（Excelの5行×2列）を横断的に処理して初めて、目視で異常に気づいた。

**修正**：`.`が数字に前後で挟まれている場合（小数点）は文区切りとみなさないよう、`splitSentences()`の正規表現を`/(?<!\d)\.(?!\d)|[。!?！？]/`へ変更した。既存の文区切り機能（全角句点、文中に無関係なASCIIピリオドがある場合等）への影響がないことも回帰確認済み。

既存25件に、「12.5 kWが破損せず抽出される」「条件(50°C)も正しく紐づく」の2件を追加し、**合計27件の自動アサーションが全件成功**することを確認した。

**教訓**：レビューでの指摘・自動アサーションの通過は「既知のテストケースに対する正しさ」しか保証しない。工程を1つ積み上げて実データを横断処理するたびに、既存の「合格」範囲の外側にあった不具合が見つかる。今後も新しい工程を追加するたびに、既存の全サンプルへ回帰的に目を通すことを運用上の注意点とする。

### 5.7 v2.4で発見・修正した設計上の不具合（工程3プロトタイプでの実データ検証で発見）

工程3のたたき台（`semantic_mapping_prototype.js`）で、`coverageGap()`をPDF要求4件・Excel設計項目5件から自動グルーピングした6つの概念グループすべてに横断適用したところ、これまで人間が手作業で選んでいた温度の1例以外で、疑わしい、あるいは明確に誤った結果が出ることが判明した。

**症状**：要求「冷房能力12 kW以上を確保すること」に対し、要求を大幅に超える設計値「冷房能力999 kW」を比較すると、明らかに要求を満たすはずが`satisfied: false`（未充足）と誤判定されていた。冷房能力・電圧・周波数・騒音の4概念グループ（6グループ中4つ）がこの誤判定の影響を受けていた。

**原因**：`coverageGap()`は「実仕様の値の範囲が、要求の範囲を覆っているか（actual ⊇ requirement）」という1方向の比較しかハードコードしていなかった。この方向は、温度のように要求・実仕様の両方が**達成可能な範囲**を表す場合（「0〜50℃で動作できること」に対し「設計は0〜40℃までしか対応しない」）には正しい。しかし冷房能力・電圧・周波数・騒音のように、要求が**片側閾値**（「12kW以上」＝下限だけを定める）で、実仕様が**単一の達成値**（設計はちょうど何kWを実現するか）である場合は、逆方向の比較（「達成値が要求の許容範囲に収まっているか＝requirement ⊇ {actual}」）が必要であり、常に破綻していた。

**検討した対応方針**：要求側の種類（範囲／閾値／厳密一致）を推定するフラグを工程4aまたは工程3に追加する案（a〜c、詳細は`semantic_mapping_prototype.md`5章に記録）を検討したが、いずれも新たな推定ロジックを要し、「推定を誤ると比較全体が誤る」というリスクを増やす。

**採用した修正**：新たな推定を一切追加せず、**実仕様（`actual`）側の`quantity`が「点」か「範囲」かを、既存の構造だけから機械的に判定**して比較方向を切り替える方式を採用した。`actual.quantity.lower.value === actual.quantity.upper.value`であれば「点」（`point_in_region`モード：達成値が要求の範囲内にあるかを判定）、そうでなければ「範囲」（`range_covers_range`モード：実仕様の範囲が要求の範囲を覆っているかを判定）とする。どちらのモードで比較したかは`comparison_mode`フィールドとして結果に明示する。この判定は新しい概念の推定を伴わない、既存データからの純粋に構造的な分岐であるため、誤推定によるリスクを増やさない。

既存27件に「999kWが12kW以上の要求を満たすと判定される」「10kWが12kW以上の要求を満たさないと判定される」「範囲vs範囲の比較で`comparison_mode`が`range_covers_range`のまま変わらない」の3件を追加し、**合計30件の自動アサーションが全件成功**することを確認した。`semantic_mapping_prototype.js`側の検出用アサーションも、修正後の正しい挙動を検証する形へ更新し、9件中9件成功を確認済み。

### 5.9 v2.6で発見・修正した設計上の不具合（外部レビューで指摘、両側区間にも波及）

### 5.10 v2.7で発見・修正した設計上の不具合（外部レビューで指摘、無限境界の包含判定）

v2.6（`comparisonMode`オプションの追加）を外部レビューへ提出したところ、片側区間・両側区間の安全停止方針と警告伝播は承認された一方、`comparisonMode`指定時の区間包含処理自体（`coversLower()`/`coversUpper()`）に、境界が無限（下限・上限なし）の場合の不具合が指摘された。指摘は着手前に実際のコードで再現を確認した（詳細は`quantity_extraction_prototype_review.md`0.6節）。

**症状**：要求「冷房能力12kW以上」（`[12, +∞)`）に対し実仕様「冷房能力0kWから20kWまで」（`[0, 20]`、20kWを超える領域は要求を満たさない）を`comparisonMode: 'actual_covers_requirement'`で比較すると、`satisfied: true`という誤った結果を返すことを確認した。

**原因**：`coversLower(outer, inner)`/`coversUpper(outer, inner)`は、`inner`側の境界が`null`（無限に広がっている）の場合、`outer`側の境界を確認せずに無条件で「覆っている」（true）を返していた。`inner`が無限に広がる場合、`outer`がそれを覆うには`outer`側も同じく無限でなければならない。

**採用した修正**：`inner`側の境界が`null`の場合は、`outer`側の境界も`null`かどうかで判定するよう修正した（`return !outer.lower`／`return !outer.upper`）。

この修正により、`coversLower`/`coversUpper`が無限境界を正しく扱えるようになったため、**片側区間と両側区間を実仕様側で区別する必要自体がなくなった**ことに気づいた。v2.5・v2.6では「実仕様が片側区間の場合は`comparisonMode`を指定しても比較不能」という制限を残していたが、この制限を撤廃し、mode指定時は片側・両側を問わず同じ経路で比較するよう統合した（mode未指定の場合の安全停止は、片側・両側とも従来どおり維持）。

レビューが提示した4件のテストケース（要求`[12,+∞)`×実仕様`[0,20]`→未充足、要求`(-∞,60]`×実仕様`[0,100]`→未充足、要求`[12,+∞)`×実仕様`[12,+∞)`→充足、要求`(-∞,60]`×実仕様`(-∞,58]`→充足）をすべて実行して確認したうえで、既存41件に追加し、**合計45件の自動アサーションが全件成功**することを確認した。`semantic_mapping_prototype.js`側はこの修正の影響を受けず、10件中10件成功のまま。

**将来課題（レビューからの軽微な改善提案、未対応）**：`lowGap`/`highGap`は`comparisonMode`によって正負の意味（不足量か余裕量か）が変わるため、将来的には`boundaryDelta`＋`deltaInterpretation`のような、モードに依存しない表現へ変更することが望ましいという指摘を受けた。今回の修正を妨げる問題ではないため、次の検討候補として残す（7章参照）。

### 5.11 v2.8で発見・修正した設計上の不具合（外部レビューで指摘、点でもmode指定が必要）

v2.7（無限境界の包含判定修正、片側・両側区間のコード統合）を外部レビューへ提出したところ、対応自体は承認されたが、「v2.6の安全策が完成」とする前にもう1点、重要な指摘を受けた。指摘は着手前に実際のコードで再現を確認した（詳細は`quantity_extraction_prototype_review.md`0.7節）。

**修正**：数値トークン`NUM`の先頭（各正規表現でスキャンが開始される最初の`NUM`）に、直前が数字またはピリオドでないことを要求する境界チェック（否定後読み`(?<![\d.])`）を追加した`NUM_AT_BOUNDARY`を導入した。2番目以降の`NUM`（範囲・公差・並列値表記の後半）は、`~`・`±`・`/`・「から」等の記号区切りが直前に必ずあるため対象外とした。この修正により、「表2.1.9200V」からは無意味な数量が一切抽出されなくなり（誤った抽出をゼロにする、安全側の変更）、正当な「400V」は引き続き正しく抽出される。回帰テスト3件を追加し、既存51件と合わせて**合計54件の自動アサーションが全件成功**することを確認した。通常の小数点付き数量（「12.5 kW」等）が境界チェック追加後も正しく抽出されることも確認済み。

### 5.14 v2.11で発見・修正した不具合、および単位辞書の拡張（公開実データ検証で発見）

5.13節の検証で「対応5単位のいずれかを含むのは16件（約15%）のみ」と記録したが、これはサンプル抽出スクリプトの単位フィルタから`mm`を書き忘れた集計ミスだった（`mm`はv2.1から対応済み）。正しくは112件中92件（約82%）が当時の対応単位を含んでいた（訂正の詳細は`semantic_mapping_prototype.md`8.19節）。

この訂正作業の過程で、実データならではの不具合をさらに2件発見・対応した。

**発見1（全角単位の不認識）**：「定格電流が15Ａ」のように、全角英字（Ａ/Ｖ等）で書かれた単位が実際の文書に現れるが、`normalizeText1to1()`は全角数字・全角ピリオドしか半角化しておらず、全角英字は認識できなかった（「２００Ｖ」が空配列を返すことを確認）。

**修正1**：全角ASCII（U+FF01-U+FF5E）はUnicode上、対応する半角ASCII（U+0021-U+007E）から一律+0xFEE0だけシフトした位置にあるため、個別の文字表ではなくオフセット計算による包括的な変換へ書き換えた。

**発見2（単位辞書拡張時のリスク）**：JIS Z 8203に準拠する圧力単位（MPa/kPa/Pa）・皮相電力単位（kVA）を追加する際、同じ実データにA（アンペア）・L（リットル）も現れていたため候補に加えたが、実際に適用したところ**Lは7件中7件、Aは4件中2件が誤検出**だった（「SUS304L」「SUS316L」というステンレス鋼種の型番、「Sn96.5Ag3.5」というはんだ成分比率の一部、「EP18A」というガラスクロス型番の一部を、単位付き数量と誤認していた）。単一アルファベット1文字の単位記号は、型番・成分比率等の数字列と衝突するリスクが実データで確認されたため、A・Lの追加は見送った（この判断は5.15節で先に予見されていた「単独のm」と同じ性質のリスクである）。

**結果**：単位辞書は5単位（℃/kW/V/Hz/dB(A)/mm）から10単位（+MPa/kPa/Pa/kVA）へ拡張された。同じ文書での対応単位を含む実文の件数は92件（約82%）から106件（約95%）へ拡大した。回帰テスト6件を追加し、既存54件と合わせて**合計59件の自動アサーションが全件成功**することを確認した。

### 5.15 単位辞書のJIS Z 8000（量及び単位）へのマスターデータ整理（v2.12）

レビューから「JIS Z 8000シリーズ（量及び単位）を、単位が同じ次元か（比較可能か）を判定する『単位ホワイトリスト』のマスターデータとして活用すべき」との指摘を受けた。JIS Z 8000は次の12部から構成される（インターネット検索により確認）：第1部（一般）・第2部（数学記号）・第3部（空間及び時間）・第4部（力学）・第5部（熱力学）・第6部（電磁気）・第7部（光及び放射）・第8部（音響学）・第9部（物理化学及び分子物理学）・第10部（原子物理学及び核物理学）・第11部（特性数）・第12部（凝縮体物理）。

**v2.12当初の対応（コメントのみ）**：`UNIT_DEFS`の各エントリが対応するJIS Z 8000の部を、コードコメントとして整理した（コードの構造自体は変更しなかった）。

**再レビューによる推奨修正（v2.12再修正）**：再レビューで「JISの部番号がコメントにしか存在せず、コードから参照・検証・出力できない。これでは『マスターデータとして整理済み』という表現には届かない」との指摘を受けた。指摘のとおりであり、`UNIT_DEFS`の各エントリへ`standard_ref: { standard, category }`を実際のフィールドとして追加し、`unitInfo()`の戻り値にも含めるよう修正した。`UNIT_DEFS`自体もモジュールのエクスポートに加え、外部から辞書全体を参照・監査できるようにした。

```js
{ source: 'MPa', canonical: 'MPa', dimension: 'pressure',
  standard_ref: { standard: 'JIS Z 8000-4', category: 'mechanics' } }
```

| `dimension` | 対応するJIS Z 8000の部 |
|---|---|
| temperature（温度） | 第5部（熱力学） |
| length（長さ） | 第3部（空間及び時間） |
| power（仕事率）・pressure（圧力） | 第4部（力学） |
| voltage（電圧）・apparent_power（皮相電力） | 第6部（電磁気） |
| frequency（周波数） | 第3部（空間及び時間。周波数は時間の逆数として定義される） |
| sound_pressure_level（音圧レベル） | 第8部（音響学） |

回帰テストを4件追加した：全エントリが`standard_ref`を持つこと、`unitInfo('MPa')`が実行時に`JIS Z 8000-4`を返すこと、同じ`dimension`（pressure）を持つMPa/kPa/Paが同じ`standard_ref`を共有すること、未知の単位では`standard_ref`が`null`になる（存在しない参照をでっち上げない）こと。既存59件と合わせて**合計63件の自動アサーションが全件成功**する。

**v2.13（軽微な将来改善への対応）**：再レビューで、「本体統合時に`UNIT_DEFS`を不変マスターとして扱うなら、`Object.freeze()`を検討できる（承認を妨げる指摘ではない）」との提案を受けた。変更コストが小さく安全性が上がるため反映した。配列自体・各エントリ・各`standard_ref`の3階層すべてを`Object.freeze()`し、外部から`UNIT_DEFS`を誤って変更できないようにした。回帰テストを1件追加し（`push()`・プロパティ代入のいずれの変更も反映されないことを確認）、既存63件と合わせて**合計64件の自動アサーションが全件成功**する。

**全角互換文字（㎡/㎥/㎜/㎏等）への対応は見送り**：JIS Z 8000はm²・m³のような合成記号を標準の表記としており、Unicode CJK互換文字ブロックの㎡・㎥等（1文字で「平方メートル」等を表す文字）はレガシー互換用途に由来する。8.18〜8.19節の実データ検証（国土交通省「公共建築工事標準仕様書」）では、対象文書がそもそも㎡・㎥等を一度も使わず、「ｍ３」のような全角の合成表記のみを用いていた（v2.11の全角ASCII正規化で対応済み）ため、CJK互換文字自体への対応は追加していない（実データでの裏付けがない拡張は行わないという8.11節以来の原則による）。将来、㎡・㎥等を含む実データが見つかった場合の設計方針として、canonical形式はJIS Z 8000に倣いm²・m³側に統一する（㎡→m²であって、その逆ではない）ことをここに記録しておく。

### 5.16 引き続き未対応（意図的な保留）

**単独の長さ単位「m」を認識しない**：「装置正面1 m」等の条件節にある「m」（メートル）は、単位テーブルに含めていないため抽出されない（`1 m`/`1m`はいずれも空配列を返すことをテストで確認済み）。「m」は他の文脈（変数名、単位以外の1文字）と衝突しやすく、安易に単位テーブルへ追加すると誤検出が増えるおそれがあるため、対応を保留している。対処案は「数字+空白+m+助詞（で/における等）」のように前後の文脈込みで判定する、または単位候補を業務ドメインごとに明示的なホワイトリストとして用意する、のいずれか。

## 6. `abstraction_levels.md`との関係の整理

本たたき台は、6工程モデルの**工程4a（単一記録内での数量抽出）のみ**を対象にしている。工程3（意味対応付け）・工程4b（文書を跨いだどの値とどの値を比較するかの決定）は依然として未着手であり、本たたき台はそれらを代替しない。

`abstraction_levels.md`8章の「実装優先順位に関する留保」で述べた「意味対応付けの**候補生成**は自動化するが、**確定**は自動化しない」という整理に対し、本たたき台は次の関係にある。

- 数量抽出結果（`unit.dimension`、`context.tokens`、`condition_candidates`）は、意味対応付けの**候補生成を補強するシグナル**として使える（例：タグが一致していても`dimension`が異なれば誤対応の疑いを立てられる）。
- v2.2のスキーマでは、`context.property`/`context.subject`/`context.state`をあらかじめ意味対応付け(工程3)用の領域として予約している（現状は常に`null`）。工程3の候補生成ロジックがこれらを埋める、という接続を想定している。
- ただし本たたき台単体では、意味対応付けの候補生成ロジックそのもの（どの`design_property`にこの数量が属するかを決める処理）は実装していない。次の検討対象になる。
- レビューの設計原則（同文書7章）「数量抽出は、数値・単位・演算子・範囲を構造化する工程であり、その値が何の設計特性を表すかを確定する工程ではない」は、本たたき台のスキーマ設計（`context.property`等をnullのまま予約する）と一致している。
