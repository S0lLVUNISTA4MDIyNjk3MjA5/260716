## 1. なぜこの順序で着手したか

`abstraction_levels.md`のレビューでは、実装優先順位として「意味対応付け（工程2/3）を数量構造化（工程4）より先に」という提案があった。一方で、現行ツール群の実装状況を見ると、

- 構造抽出（章・節、Excel行列）：**実装済み**、検証済み
- タグ付け・照合エンジン（exact/code/tag/vector/synonym/fuzzy、確信度階層）：**実装済み**、検証済み
- 数量・単位の抽出と比較：**未着手**（現状は`trace_text`という1本の文字列の一部として埋もれており、構造化されたことが一度もない）

という非対称がある。他の工程は「改良ベースで機能アップ」できる状態にあるのに対し、数量抽出だけは概念（`abstraction_levels.md`の工程4）のまま何も検証されていなかった。そこで、**実装ロードマップの優先順位を変更するのではなく**、まず数量抽出を「同じ土俵」に乗せる目的で、実データを使った動くたたき台を作成した。

## 2. スキーマ（たたき台）

抽出結果は次の形の`Quantity`オブジェクトとする。

```js
{
  raw_text: "0 °C～40 °C",           // 元テキストの一致部分
  value: {
    kind: "single" | "range" | "alternatives",
    amount: 40,                      // kind:"single"のとき
    min: 0, max: 40,                 // kind:"range"のとき
    options: [50, 60],               // kind:"alternatives"のとき(例: "50/60 Hz")
  },
  unit: { raw: "°C", canonical: "degC", category: "temperature" },
  operator: "exact" | "at_least" | "at_most" | "less_than" | "more_than" | "range" | "alt" | "..._approx",
  condition: { raw_text: "40 °C", unit: "degC", value: {...} } | undefined, // この値が成立する前提条件(あれば)
  needs_review: "..." | undefined,   // 抽出時に矛盾・曖昧性を検知した場合の理由
}
```

`condition`フィールドが本たたき台の要点である。「周囲温度50 °Cにおいて、冷房能力12 kW以上」のような複文から、後述のとおり「12kW以上」という値が「周囲温度50°C」という条件付きであることを分離して保持する。条件を無視して数値だけ比較すると、異なる前提条件下の値を誤って比較してしまう（4章参照）。

## 3. 実データでの検証結果

`samples/hvac_trace_sample_small/`の実文14件（PDF要求4件＋Excel標準機種情報/検討結果 各5件）を対象に抽出した。全件の抽出結果は`quantity_extraction_prototype.js`を実行すれば再現できる。代表例のみ示す。

| 原文 | 抽出結果（要約） |
|---|---|
| 「空調ユニットは、周囲温度0 °Cから50 °Cの環境で正常に運転できること。」 | `range(0, 50, degC)` |
| 「周囲温度50 °Cにおいて、冷房能力12 kW以上を確保すること。」 | `single(50, degC)` ＋ `single(12, kW, operator=at_least, condition=50°C)` |
| 「定格電源は三相AC 220 V、50 Hzとすること。」 | `single(220, V)` ＋ `single(50, Hz)` |
| 「三相AC 200 V、50/60 Hz」 | `single(200, V)` ＋ `alternatives([50, 60], Hz)` |
| 「0 °C～40 °C」 | `range(0, 40, degC)` |

## 4. 工程5（比較・横断分析）への橋渡し：温度範囲の充足判定

抽出したPDF要求とExcel側の値を実際に比較し、`abstraction_levels.md`4章で示した「関連性・標準適合性・対応後適合性を分離する」の温度例を、コードで再現した。

```text
PDF要求:        range(0, 50, degC)
Excel標準機種:   range(0, 40, degC)
Excel検討結果:   range(0, 50, degC)

要求 vs 標準機種: satisfied=false, highGap=10   → 「10℃不足」
要求 vs 検討結果: satisfied=true,  highGap=0    → 「充足」
```

これは、①数量を構造化して取り出し、②単位が一致するか確認し、③範囲同士の被覆判定を計算する、という3手順だけで得られた結果であり、AIを一切使用していない。`abstraction_levels.md`で述べた「工程4（抽出）は非AIで可能」という主張を、実データで裏付けた。

**ただし注意点**：この比較は「PDF要求の`range(0,50)`」と「Excel標準機種の`range(0,40)`」が**同じ設計特性を指している**という前提のもとでのみ成立する。この前提の判定自体（工程3・意味対応付け）は、本たたき台の範囲外であり、今回は人間が読んで対応させた2つの文字列を直接渡している。数量抽出（工程4a）は意味対応付け（工程3）の代替にはならず、あくまで工程3が終わった後（または既存のタグ一致候補に対する補強シグナルとして）機能するものである。

## 5. 発見した不具合・限界

たたき台を実際に動かしたことで、設計段階では気づかなかった問題が3件見つかった。「概念だけでは分からない、実装して初めて分かること」の実例として記録する。

### 5.1 修正済み：範囲表現で単位が両側にある形式を取りこぼしていた

初版の正規表現は`X~Y unit`（単位1回）だけを想定していたが、実データの大半は`0 °C～40 °C`のように**単位が両側にある**表記だった。これは本たたき台のフラグシップ例（温度範囲の比較）そのものであり、修正しなければ4章の比較デモが成立しなかった。正規表現を単位0回・1回どちらにも対応するよう修正し、単位表記が前後で食い違う場合は`needs_review`フラグを立てるようにした。

### 5.2 未対応：単独の長さ単位「m」を認識しない

「装置正面1 mにおける騒音値は60 dB(A)以下」「装置正面1 mで65 dB(A)」という条件節にある**「m」（メートル）**は、単位テーブルに含めていないため抽出されず、`60 dB(A)`側の`condition`が付かないまま出力される。

意図的な保留である。「m」は他の文脈（変数名、単位以外の1文字）と衝突しやすく、安易に単位テーブルへ追加すると誤検出が増えるおそれがある。対処案は「数字+空白+m+助詞（で/における等）」のように前後の文脈込みで判定する、または単位候補を業務ドメインごとに明示的なホワイトリストとして用意する、のいずれか。今回はこの判断を保留し、既知の限界として記録するに留めた。

### 5.3 未対応：条件節の判定が単純な直後マーカーに依存している

「〜において」「〜で」という語が直後にあれば次の数量の`condition`にする、という単純なルールで実装した。実データではたまたま機能したが、次のようなケースは想定していない。

- 条件が数量の**後**に来る文（「12kW以上を確保すること。ただし周囲温度50℃時。」等）
- 条件節と要求節の間に無関係な修飾語が挟まる文
- 1つの条件が複数の数量にかかる文

これらは本たたき台の対象外とし、実データがもっと増えた段階で再設計が必要になる可能性がある。

## 6. `abstraction_levels.md`との関係の整理

本たたき台は、6工程モデルの**工程4a（単一記録内での数量抽出）のみ**を対象にしている。工程3（意味対応付け）・工程4b（文書を跨いだどの値とどの値を比較するかの決定）は依然として未着手であり、本たたき台はそれらを代替しない。

`abstraction_levels.md`8章の「実装優先順位に関する留保」で述べた「意味対応付けの**候補生成**は自動化するが、**確定**は自動化しない」という整理に対し、本たたき台は次の関係にある。

- 数量抽出結果（`quantity_type`＝`category`、`condition`）は、意味対応付けの**候補生成を補強するシグナル**として使える（例：タグが一致していても`category`が異なれば誤対応の疑いを立てられる）。
- ただし本たたき台単体では、意味対応付けの候補生成ロジックそのもの（どの`design_property`にこの数量が属するかを決める処理）は実装していない。次の検討対象になる。

## 7. 次の検討候補

1. 5.2・5.3で保留した限界への対処方針（単位ホワイトリストの設計、条件節判定の高度化）
2. 抽出した数量を、既存の照合結果（タグ一致候補）に**補強シグナルとして統合**する設計（`category`不一致を「要確認」の追加根拠にする等）
3. 単位変換（例：`℉`→`degC`、複合単位）が必要になった場合の設計
4. この抽出ロジックを、実際にツール（Excel→JSON／PDF→JSON／照合ツール）へ組み込む場合の実装場所（変換時に付加するか、照合時にオンザフライで抽出するか）

いずれも未着手・未実装であり、実装の是非を含めて議論中である。
