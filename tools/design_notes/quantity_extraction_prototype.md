## 1. なぜこの順序で着手したか

`abstraction_levels.md`のレビューでは、実装優先順位として「意味対応付け（工程2/3）を数量構造化（工程4）より先に」という提案があった。一方で、現行ツール群の実装状況を見ると、

- 構造抽出（章・節、Excel行列）：**実装済み**、検証済み
- タグ付け・照合エンジン（exact/code/tag/vector/synonym/fuzzy、確信度階層）：**実装済み**、検証済み
- 数量・単位の抽出と比較：**未着手**（現状は`trace_text`という1本の文字列の一部として埋もれており、構造化されたことが一度もない）

という非対称がある。他の工程は「改良ベースで機能アップ」できる状態にあるのに対し、数量抽出だけは概念（`abstraction_levels.md`の工程4）のまま何も検証されていなかった。そこで、**実装ロードマップの優先順位を変更するのではなく**、まず数量抽出を「同じ土俵」に乗せる目的で、実データを使った動くたたき台を作成した。

## 2. スキーマ（v2.1、レビューの推奨スキーマを採用）

初版は`value.kind: "single"|"range"|"alternatives"`という3種類の形を持っていたが、レビュー4章の指摘に従い、単一値・片側制約・範囲を**区間（interval）モデルへ統一**した。単一値は`lower === upper`の退化区間として表現する。

```js
{
  source_text: "５０℃～６０℃",       // 元テキストそのもの(全角等を正規化していない)
  normalized_text: "50℃~60℃",       // 正規化後の文字列(全角→半角、～→~等)
  quantity: {
    kind: "interval" | "alternatives",
    lower: { value: 50, inclusive: true } | null,   // 片側のみの制約(以下/未満)ならnull
    upper: { value: 60, inclusive: true } | null,   // 片側のみの制約(以上/超える)ならnull
    options: [50, 60],               // kind:"alternatives"のとき(例: "50/60 Hz")
    selection_semantics: "unknown",  // alternativesの意味(択一/両対応等)は確定しない
  },
  unit: { source: "℃", canonical: "degC", dimension: "temperature" },
  qualifiers: [{ type: "approximate" | "maximum" | "minimum" | "nominal", source_text: "約" }], // あれば
  context: {
    property: null, subject: null, state: null,   // 工程3(意味対応付け)が埋める領域。本工程では確定しない
    tokens: ["AC", "三相"],           // 周辺語(定格/交流/直流等)をそのまま保持
  },
  condition_candidates: [{ source_text: "50 °C", quantity: {...}, unit: {...}, confidence: 0.7 }], // あれば
  extraction: { confidence: 0.95, warnings: [] },
}
```

`condition_candidates`（複数形・確信度付き）が本たたき台の要点である。「周囲温度50 °Cにおいて、冷房能力12 kW以上」のような複文から、「12kW以上」という値が「周囲温度50°C」という条件付きであることを分離して保持する。条件を無視して数値だけ比較すると、異なる前提条件下の値を誤って比較してしまう（4章参照）。v2では**同一文内に限定**して条件候補を付与し、句点をまたいだ誤伝播を防いでいる（5.1節参照）。

## 3. 実データでの検証結果

`samples/hvac_trace_sample_small/`の実文14件（PDF要求4件＋Excel標準機種情報/検討結果 各5件）に加え、レビューが提示した境界系・失敗系テストケース21件（5章参照）を対象に抽出した。全件の抽出結果は`quantity_extraction_prototype.js`を実行すれば再現できる。代表例のみ示す（v2のスキーマで表記）。

| 原文 | 抽出結果（要約） |
|---|---|
| 「空調ユニットは、周囲温度0 °Cから50 °Cの環境で正常に運転できること。」 | `interval(lower=0, upper=50, degC)` |
| 「周囲温度50 °Cにおいて、冷房能力12 kW以上を確保すること。」 | `interval(exact 50, degC)` ＋ `interval(lower=12 incl, upper=null, kW, condition_candidates=[50°C])` |
| 「定格電源は三相AC 220 V、50 Hzとすること。」 | `interval(exact 220, V, tokens=[三相,AC])` ＋ `interval(exact 50, Hz)` |
| 「三相AC 200 V、50/60 Hz」 | `interval(exact 200, V, tokens=[三相,AC])` ＋ `alternatives([50, 60], Hz, selection_semantics=unknown)` |
| 「0 °C～40 °C」 | `interval(lower=0, upper=40, degC)` |
| 「0℃以上50℃以下」（レビュー提示） | `interval(lower=0 incl, upper=50 incl, degC)`（片側制約2件から1件へ統合） |
| 「-10～50℃」（レビュー提示） | `interval(lower=-10, upper=50, degC)`（符号付き数値） |
| 「50℃で停止する。電源は220Vとする」（レビュー提示） | `interval(exact 50, degC)` ＋ `interval(exact 220, V)`（**文をまたぐため条件は伝播しない**） |
| 「1,500 mm」（追加発見） | `interval(exact 1500, mm)` |
| 「50±2℃」（追加発見） | `interval(lower=48, upper=52, degC, warnings=[公差表記から算出])` |

## 4. 工程5（比較・横断分析）への橋渡し：温度範囲の充足判定

抽出したPDF要求とExcel側の値を実際に比較し、`abstraction_levels.md`4章で示した「関連性・標準適合性・対応後適合性を分離する」の温度例を、コードで再現した。v2では、レビュー2.5の指摘に従い比較結果が**暫定判定であること自体を構造化して明示**する。

```text
PDF要求:        interval(lower=0, upper=50, degC)
Excel標準機種:   interval(lower=0, upper=40, degC)
Excel検討結果:   interval(lower=0, upper=50, degC)

要求 vs 標準機種: { comparable:true, provisional:true, satisfied:false, highGap:10, assumptions:[...] } → 「10℃不足」
要求 vs 検討結果: { comparable:true, provisional:true, satisfied:true,  highGap:0,  assumptions:[...] } → 「充足」
```

`assumptions`には「同じ設計特性として選択済み」「同じ運転条件」「単位換算不要」「抽出結果に要確認事項なし」という、この比較が成立するための前提が明示される。これらの前提が崩れている場合（設計特性が実は違う、運転条件が違う等）、比較結果は無条件に信頼してはならないことをデータ構造自体で示す。

これは、①数量を構造化して取り出し、②単位が一致するか確認し、③区間同士の被覆判定を計算する、という3手順だけで得られた結果であり、AIを一切使用していない。`abstraction_levels.md`で述べた「工程4（抽出）は非AIで可能」という主張を、実データで裏付けた。

**ただし注意点**：この比較は「PDF要求の`range(0,50)`」と「Excel標準機種の`range(0,40)`」が**同じ設計特性を指している**という前提のもとでのみ成立する。この前提の判定自体（工程3・意味対応付け）は、本たたき台の範囲外であり、今回は人間が読んで対応させた2つの文字列を直接渡している。数量抽出（工程4a）は意味対応付け（工程3）の代替にはならず、あくまで工程3が終わった後（または既存のタグ一致候補に対する補強シグナルとして）機能するものである。

## 5. 発見した不具合・限界

### 5.1 初版で発見・修正済みの不具合

初版たたき台を実際に動かしたことで、設計段階では気づかなかった問題が見つかった。「概念だけでは分からない、実装して初めて分かること」の実例として記録する。

**範囲表現で単位が両側にある形式を取りこぼしていた**：初版の正規表現は`X~Y unit`（単位1回）だけを想定していたが、実データの大半は`0 °C～40 °C`のように**単位が両側にある**表記だった。これは本たたき台のフラグシップ例（温度範囲の比較）そのものであり、修正しなければ4章の比較デモが成立しなかった。

### 5.2 レビューで発見され、v2で修正済みの不具合

`quantity_extraction_prototype_review.md`によるレビューで、初版になお残っていた問題が6件指摘された。全件をコードで再現・検証したうえで修正した（検証方法・結果は同レビュー文書の「対応記録」節を参照）。

| # | 問題 | 修正内容 |
|---|---|---|
| 1 | 符号付き数値（`-10℃`等）を抽出できない | ASCIIハイフン・全角マイナス・Unicodeマイナスを正規化し、符号付き数値に対応 |
| 2 | `0℃以上50℃以下`が2つの独立した単一値になる | 片側制約の隣接ペアを1つの区間へ統合するロジックを追加 |
| 3 | 境界の包含・非包含（以上/以下 vs 未満/超える）を保持していない | `{value, inclusive}`構造へ変更 |
| 4 | `raw_text`が正規化後の文字列で、真の原文ではない | 正規化を1:1文字置換に限定し、`source_text`（原文）と`normalized_text`（正規化後）を分離して保持 |
| 5 | 比較結果（`coverageGap()`）が暫定判定であることを構造化していない | `provisional: true`と`assumptions`配列を明示 |
| 6 | 条件節（`condition`）が文をまたいで誤伝播する（`50℃で停止する。電源は220Vとする`で実際に再現した） | 文単位に分割してから条件候補を付与するよう変更。かつ`condition`→`condition_candidates`（複数・確信度付き）へ変更し、自動確定ではなく候補として保持 |

推奨修正（3.1〜3.4）についても、単位比較のcanonical化、スラッシュ表現の意味を確定しない（`selection_semantics: "unknown"`）、周辺語(`context.tokens`)の保持、逆転範囲の検出を反映した。

### 5.3 レビュー対応中に追加で発見した不具合（レビュー未記載）

v2の実装・テスト過程で、レビューが明示的には挙げていなかった2件の不具合を独自に発見し、修正した。いずれも「`needs_review`フラグすら立たず、黙って誤った数値を返す」という、フラグ付きの不確実性より危険な種類の不具合だった。

- **`50±2℃`のような公差表記で、中心値（50）が失われ公差（2）だけが抽出される**：`±`を考慮しない数値正規表現が「2℃」だけにマッチしていた。修正後は`interval(lower=48, upper=52)`として区間化し、公差由来であることを`warnings`に記録する。
- **`1,500 mm`のような桁区切りカンマで、「1」が失われ「500 mm」になる**：数値正規表現がカンマ区切りを考慮していなかった。修正後はカンマ区切りを許容する数値トークンへ変更した。

### 5.4 v2.1で追加対応した事項

v2に対する再レビューで、スキーマには境界の包含・非包含が存在する一方、`coverageGap()`が数値だけを比較し、境界が等しい場合の`inclusive`を考慮していないことが判明した。また、`約50℃`・`最大50℃`のような修飾語が`source_text`から外れ、`最大50℃`が完全一致値`[50,50]`として扱われていた。

v2.1では次を追加した。

- 要求境界を含むが実仕様境界が含まない場合、被覆不足と判定する。
- 片側が無限区間の場合を明示的に処理する。
- 数値差が0でも境界包含が異なる場合を`boundaryMismatch`で示す。
- `約`・`およそ`・`程度`・`最大`・`最小`・`公称`を`qualifiers`として保持する。
- 修飾語を`source_text`／`normalized_text`の抽出範囲へ含める。
- `最大`は上限候補、`最小`は下限候補として抽出し、文脈による確定が必要であることを警告する。
- 既存18件に、修飾語3件と境界被覆2件を追加し、合計23件の自動アサーションが全件成功することを確認した。

### 5.5 引き続き未対応（意図的な保留）

**単独の長さ単位「m」を認識しない**：「装置正面1 m」等の条件節にある「m」（メートル）は、単位テーブルに含めていないため抽出されない（`1 m`/`1m`はいずれも空配列を返すことをテストで確認済み）。「m」は他の文脈（変数名、単位以外の1文字）と衝突しやすく、安易に単位テーブルへ追加すると誤検出が増えるおそれがあるため、対応を保留している。対処案は「数字+空白+m+助詞（で/における等）」のように前後の文脈込みで判定する、または単位候補を業務ドメインごとに明示的なホワイトリストとして用意する、のいずれか。

## 6. `abstraction_levels.md`との関係の整理

本たたき台は、6工程モデルの**工程4a（単一記録内での数量抽出）のみ**を対象にしている。工程3（意味対応付け）・工程4b（文書を跨いだどの値とどの値を比較するかの決定）は依然として未着手であり、本たたき台はそれらを代替しない。

`abstraction_levels.md`8章の「実装優先順位に関する留保」で述べた「意味対応付けの**候補生成**は自動化するが、**確定**は自動化しない」という整理に対し、本たたき台は次の関係にある。

- 数量抽出結果（`unit.dimension`、`context.tokens`、`condition_candidates`）は、意味対応付けの**候補生成を補強するシグナル**として使える（例：タグが一致していても`dimension`が異なれば誤対応の疑いを立てられる）。
- v2.1のスキーマでは、`context.property`/`context.subject`/`context.state`をあらかじめ意味対応付け(工程3)用の領域として予約している（現状は常に`null`）。工程3の候補生成ロジックがこれらを埋める、という接続を想定している。
- ただし本たたき台単体では、意味対応付けの候補生成ロジックそのもの（どの`design_property`にこの数量が属するかを決める処理）は実装していない。次の検討対象になる。
- レビューの設計原則（同文書7章）「数量抽出は、数値・単位・演算子・範囲を構造化する工程であり、その値が何の設計特性を表すかを確定する工程ではない」は、本たたき台のスキーマ設計（`context.property`等をnullのまま予約する）と一致している。

## 7. 次の検討候補

1. 5.5で保留した限界への対処方針（単位ホワイトリストの設計）
2. 条件候補判定の高度化（現状は同一文内・直後マーカーのみ。条件が数量の後に来る文、条件節と要求節の間に修飾語が挟まる文、1つの条件が複数の数量にかかる文には未対応）
3. 抽出した数量を、既存の照合結果（タグ一致候補）に**補強シグナルとして統合**する設計（`dimension`不一致を「要確認」の追加根拠にする等）
4. 単位変換（例：`℉`→`degC`、複合単位）が必要になった場合の設計
5. この抽出ロジックを、実際にツール（Excel→JSON／PDF→JSON／照合ツール）へ組み込む場合の実装場所（変換時に付加するか、照合時にオンザフライで抽出するか）

いずれも未着手・未実装であり、実装の是非を含めて議論中である。
