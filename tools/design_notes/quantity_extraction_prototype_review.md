## 0.2 独自の追加検証と修正（v2.2、2026-07-18）

v2.1の受領後、既存アサーションではカバーされない組み合わせを独自に追加検証した。「0℃以上50℃以下」（`以上`/`以下`）は1件の区間へ統合される一方、「最小0℃最大50℃」（`最大`/`最小`）は確信度0.6の片側候補2件のまま統合されないという非対称な挙動を発見した。コード上の原因（`_boundSide`が`最大`/`最小`由来のとき設定されず、隣接統合ロジックの対象外になっていた）は明確だったが、これが意図的な安全設計か実装漏れかがコード・文書のいずれにも記載されていなかったため、対応方針を確認した。

方針: `以上`/`以下`と同様に統合する。`_boundSide`を`最大`/`最小`由来でも設定するよう修正し、隣接する片側境界を区間へ統合する。統合後も、各片側候補が持っていた`warnings`（「文脈による確定が必要」等）は引き継がれるため、統合後の区間も確信度は自動的に低いまま（0.6）保たれる。

既存23件に、「最小0℃最大50℃」の統合と、統合後も確信度が0.6のままであることを検証する2件を追加し、**合計25件中25件成功**を確認した。単独で現れる`最大50℃`・`最小10kW`（隣接する反対側の境界がない場合）が誤って無関係な他の数量と結合されないことも回帰確認済み。

工程4aは、現時点の対象範囲に対する基準版として一旦固定してよい段階と判断する。ただし、限界の表現は次のように限定する。

> 現在定義した単位・数量表現・テスト範囲における既知の限界は、単独の`m`単位を認識しないことである。

これは、将来の単位変換、複合単位、OCR誤認識、後置条件などまで解決済みという意味ではない。

### 次工程の推奨

次は、条件節判定を文法規則だけで高度化するより先に、工程3「意味対応付けの候補生成」へ進むことを推奨する。

理由は、数量の役割が分からない状態で条件規則を増やすと、次の値を安全に区別できないためである。

- `50℃`: 周囲使用温度の条件値
- `12kW`: 冷房能力の要求値
- `1m`: 騒音測定距離の条件値
- `60dB(A)`: 運転騒音の上限値

工程3の最小プロトタイプは、数量抽出結果と周辺文脈から、設計特性と数量の役割を**確定せず候補提示**する。

```json

    {
      "concept_id": "environment.ambient_operating_temperature",
      "quantity": "50℃",
      "role": "condition",
      "confidence": 0.91,
      "evidence": ["周囲温度", "temperature"]

      "evidence": ["冷房能力", "power"]

```

候補生成では、少なくとも次を根拠として保持する。

- 周辺語
- 単位次元
- 章・節
- Excel設計項目名
- タグ
- 除外語
- 数量間の位置関係

## 0.4 レビュアーコメント：v2.4比較方向修正への指摘と安全策の適用（v2.5、2026-07-18）

**指摘の要旨:** v2.4は「actualが点か範囲か」の2値だけで比較方向を決めていたが、これでは「達成点」と「対応可能領域」の2種類しか区別できない。実データには少なくとも次の5種類の区間の意味があり、v2.4はこのうち後者3種類（保証下限・保証上限・変動範囲）を扱えない。

| 区間の意味 | 例 | 必要な比較 |
|---|---|---|
| 対応可能領域 | 使用温度0〜50℃ | actualが要求領域を包含 |
| 達成値 | 冷房能力12.5kW | 点が要求条件内か |
| 保証下限 | 冷房能力15kW以上 | actual下限が要求下限以上か |
| 保証上限 | 騒音58dB以下 | actual上限が要求上限以下か |
| 変動・公差範囲 | 220V±5% | actual全体が要求許容領域内か |

具体的には、要求「騒音60dB(A)以下」に対し実仕様「最大58dB(A)」（明らかに充足）が`satisfied:false`、要求「冷房能力12kW以上」に対し実仕様「最小15kW」（明らかに充足）が`satisfied:false`となることを、実際に`coverageGap()`を実行して確認した。

あわせて次の2件も再現・確認した。

- 空集合の誤認：`50℃以上50℃未満`（`[50,50)`、値を1つも含まない矛盾した区間）が、値の一致だけを見る旧`isPoint`相当の判定により「点50℃」として扱われ、誤った比較結果を返す。
- 警告の握りつぶし：`最大58dB(A)`等、`extraction.warnings`に「文脈による確定が必要」という警告が付いている値でも、`coverageGap()`の`assumptions`は常に固定文字列「抽出結果に要確認事項なし」を返し、警告が比較結果へ伝播していなかった。

**対応方針:** レビューが提案した恒久対応（工程3が区間の意味候補`interval_semantics`を生成し、比較モードを明示的に選択する設計）は、工程3の設計自体を拡張する必要があり規模が大きいため、次のステップとして別途着手することとし、まずレビューが提示した「安全な暫定修正」をv2.5として先に反映した。

- actualが真の点（値が一致し、かつ両端がinclusive。空集合を誤って点扱いしない）：`point_in_region`のまま許可。
- actualが両側区間（下限・上限の両方を持つ）：`range_covers_range`のまま許可。要求側が実質1点（例：220V）であっても、通常の範囲比較として正しく動くことを確認した。
- actualが片側区間（下限・上限の片方のみ）：**自動で比較方向を決めず、`comparable:false`＋理由を返す。** これは「保証下限」「保証上限」のいずれかを実仕様の構造だけからは一意に判定できないため。誤った`satisfied:false`を返すより、「比較方向を確定できない」と返す方が安全という判断に同意した。
- 空集合（`[50,50)`等）は`comparable:false`を返す。
- 抽出時の警告（`requirement`/`actual`双方）を`extractionWarnings`として比較結果へ含める。固定文字列の「抽出結果に要確認事項なし」は削除した。

## 0.5 レビュアーコメント：両側区間の意味も未確定という追加指摘（v2.6、2026-07-18）

v2.5（片側区間は自動判定せず比較不能を返す安全策）を提出したところ、次の追加指摘を受けた。指摘は着手前に全件、実際のコードで再現するかを検証した（推測で対応せず、レビューの主張を裏取りした上で着手）。**指摘は全て実際に再現した。**

**指摘の要旨**：v2.5は片側区間だけを安全策の対象にしたが、**actualが両側区間の場合も、その区間が何を意味するかは一意に決まらない**。両側区間には少なくとも次の異なる意味がある。

- 対応可能領域：使用温度0〜50℃
- 許容領域：電圧200〜240Vを許容
- 実現値の変動範囲：電圧220±10V
- 測定結果のばらつき：58〜60dB
- 保証範囲：能力12〜15kW

v2.5は両側区間を無条件で「実仕様が要求範囲を覆っているか（actual ⊇ requirement）」の方向（`range_covers_range`）で判定していた。これは温度のような「対応可能領域」には正しいが、公差表記のような「変動範囲」には逆方向が必要になる。具体的には、要求「電圧200〜240V」に対し実仕様「電圧220±10V」（実質210〜230V、要求の許容範囲内に収まる変動）は、変動範囲として解釈すれば充足のはずだが、v2.5は`satisfied:false`を返すことを確認した。逆に、要求「電圧220V」（厳密値）に対し同じ実仕様「電圧220±10V」は、変動する可能性がある以上「充足と断定できない」はずが、v2.5は`satisfied:true`を返すことも確認した。

あわせて、既に対応済みの3項目（片側区間の安全策、空区間の誤認防止、警告伝播）についても再現確認を受け、いずれも承認された。ただし警告伝播については、`coverageGap()`の早期return経路（単位不一致・非interval形式チェック）が警告収集より前に実行されるため、これらの経路では警告が伝播しないという実装順序の不備を指摘され、これも再現・確認した。

**対応方針**：レビューが提案した「`coverageGap()`に第3引数`options.comparisonMode`を追加し、両側区間もmodeが明示されない限り比較不能を返す」という設計をそのまま採用した。

- `comparisonMode: 'actual_covers_requirement'`：actualが対応可能領域を表す場合。実仕様の範囲が要求範囲を覆っているかを判定する（v2.5までの`range_covers_range`と同じロジック）。
- `comparisonMode: 'requirement_covers_actual'`：actualが変動・公差範囲や測定結果のばらつきを表す場合。実仕様の範囲全体が要求の許容範囲に収まっているかを判定する（新規追加。要求側とactual側を入れ替えた被覆判定）。
- modeが指定されない場合：`comparable:false`＋理由を返す。
- 警告収集（`extractionWarnings`の計算）を関数冒頭へ移動し、単位不一致・非interval形式の早期return経路でも警告が保持されるようにした。

## 0.6 レビュアーコメント：無限境界の包含判定が逆という指摘（v2.7、2026-07-18）

v2.6（`comparisonMode`オプションの追加）を提出したところ、2件の対応方針（mode未指定時の両側区間の安全停止、警告収集の早期return対応）は承認された一方、`comparisonMode`指定時の区間包含処理（`coversLower()`/`coversUpper()`）自体に、境界が無限（下限・上限なし）の場合の不具合が1件指摘された。指摘は着手前に実際のコードで再現を確認した（推測で対応せず、レビューの主張を裏取りした上で着手）。**指摘は実際に再現した。**

**症状**：要求「冷房能力12kW以上」（`[12, +∞)`）に対し、実仕様「冷房能力0kWから20kWまで」（`[0, 20]`、集合として要求を包含しない。20kWを超える領域が要求されているのに実仕様は対応していない）を`comparisonMode: 'actual_covers_requirement'`で比較すると、`satisfied: true`という誤った結果を返すことを確認した。

**原因**：`coversLower(outer, inner)`/`coversUpper(outer, inner)`は、`inner`側（このケースでは要求）の境界が`null`（無限に広がっている）の場合、無条件で`true`（覆っている）を返していた。しかし、`inner`が無限に広がっている場合、`outer`（このケースでは実仕様）がそれを覆うには`outer`側の境界も`null`（同じく無限）でなければならない。有限の実仕様`[0,20]`は、無限に広がる要求`[12,+∞)`を決して覆えない。

**採用した修正**：`coversLower()`/`coversUpper()`を、`inner`側の境界が`null`の場合は`outer`側の境界も`null`かどうかで判定するよう修正した（レビュー提案の実装をそのまま採用）。

```js
function coversLower(outer, inner) {
  if (!inner.lower) return !outer.lower;
  if (!outer.lower) return true;
  // ...(以下不変)
}
```

あわせて、この修正により`coversLower`/`coversUpper`が無限境界を正しく扱えるようになったため、**片側区間と両側区間を区別する必要自体がなくなった**ことに気づいた。v2.6までは「実仕様が片側区間の場合は`comparisonMode`を指定しても比較不能」という制限を残していたが、無限境界の扱いが正しくなった以上、`comparisonMode`が明示されていれば片側区間の実仕様も同じロジックで正しく比較できる（例：要求`[12,+∞)`と実仕様`[12,+∞)`が一致するかどうかも判定可能になった）。この制限を撤廃し、mode指定時は片側・両側を問わず同じ経路で比較するよう統合した（mode未指定の場合の安全停止は、片側・両側とも従来どおり維持）。

レビューが提示した4件の追加テストケース（要求`[12,+∞)`×実仕様`[0,20]`→未充足、要求`(-∞,60]`×実仕様`[0,100]`→未充足、要求`[12,+∞)`×実仕様`[12,+∞)`→充足、要求`(-∞,60]`×実仕様`(-∞,58]`→充足）を、すべて実際に`coverageGap()`を実行して期待どおりの結果になることを確認したうえで、既存41件に追加し、**合計45件の自動アサーションが全件成功**することを確認した（v2.7、`quantity_extraction_prototype.js`）。`semantic_mapping_prototype.js`側はこの修正の影響を受けず、10件中10件成功のまま。詳細は`quantity_extraction_prototype.md`5.10節を参照。

**レビューからの軽微な改善提案（未対応・将来課題）**：`lowGap`/`highGap`は`comparisonMode`によって正負の意味（不足量か余裕量か）が変わるため、将来的には`boundaryDelta`＋`deltaInterpretation`のような、モードに依存しない表現へ変更することが望ましいという指摘を受けた。レビュー自身も「今回の修正を止める問題ではない」としており、本ラウンドでは対応せず、次の検討候補として残す（`quantity_extraction_prototype.md`7章参照）。

## 0.7 レビュアーコメント：点でもcomparisonModeが必要という指摘（v2.8、2026-07-18）

v2.7（無限境界の包含判定修正、片側・両側区間のコード統合）は承認されたが、「v2.6の安全策が完成」とする前にもう1点、重要な指摘を受けた。指摘は着手前に実際のコードで再現を確認した（推測で対応せず、レビューの主張を裏取りした上で着手）。**指摘は実際に再現した。**

**指摘の要旨**：`actual`が真の点であっても、意味は一意ではない。要求「使用温度0〜50℃」に対し実仕様「使用温度25℃」の場合、25℃が「試験で確認した1点（達成値）」なら要求範囲内の点として適合するが、「対応可能な温度が25℃だけ（能力領域）」という意味なら、0〜50℃の要求範囲を覆っていないため未充足になる。同じ数値でも解釈で結果が逆転する。しかしv2.7までは、`actual`が点であれば`comparisonMode`の指定を無視して常に`point_in_region`へ入っていたため、`coverageGap(requirement, actual, { comparisonMode: 'actual_covers_requirement' })`のように明示的に渡しても無視され、`comparison_mode`が`point_in_region`のまま`satisfied:true`を返すことを確認した。これはAPI利用者にとって予想外の挙動である。

**採用した対応**：レビュー提案の3モード設計をそのまま採用した。

- `comparisonMode`が明示された場合：`actual`が点か区間かにかかわらず、指定されたモードで比較する。`point_in_region`が明示された場合は`actual`が真の点であることを要求し（点でなければ`comparable:false`）、`actual_covers_requirement`/`requirement_covers_actual`が明示された場合は、点を退化区間`[v,v]`とみなし、v2.7で修正済みの`coversLower`/`coversUpper`による一般的な区間包含判定へそのまま合流させる。
- `comparisonMode`が未指定かつ`actual`が真の点：暫定的に`point_in_region`として扱う（既存デモ・回帰テストとの互換性を優先し、レビューが提示した2案「暫定的にpoint_in_region」「完全な安全策なら比較不能」のうち軽い方を採用した）。
- `comparisonMode`が未指定かつ`actual`が非点区間：従来どおり比較不能（v2.5〜v2.7から変更なし）。

レビューが提示した4件のテストケース（要求`[0,50]`×点`25`、mode=`point_in_region`→充足／mode=`actual_covers_requirement`→未充足／mode=`requirement_covers_actual`→充足／`comparison_mode`が指定値と一致することの確認）を、すべて実際に`coverageGap()`を実行して期待どおりの結果になることを確認したうえで、既存45件に追加し、**合計49件の自動アサーションが全件成功**することを確認した（v2.8、`quantity_extraction_prototype.js`）。`semantic_mapping_prototype.js`側はこの修正の影響を受けず、10件中10件成功のまま（本プロトタイプは`point_in_region`をmode未指定のまま使う既存の呼び出し方のみで、mode明示の新しい分岐は使用していないため）。

**レビューの結論**：区間の形（片側・両側・点）だけでは意味を決められないという原則は、点にも当てはまる。次の`interval_semantics`候補生成では、少なくとも`achieved_point`/`capability_domain`/`acceptable_region`/`outcome_range`/`guaranteed_minimum`/`guaranteed_maximum`を扱う必要があり、その候補から`comparisonMode`を選択する設計へ進むのが妥当と評価された。「点に対する明示モードの尊重」を修正した本ラウンドをもって、比較エンジン側（`coverageGap()`）の基盤は一旦固定してよいとの判断を受けた。

> **注記:** 以下の1〜5章は初回レビュー時点の指摘内容を保存した履歴である。現在の対応状況は0〜0.7節および6章を参照する。

## 1. 総合評価

本プロトタイプは、数量抽出を実データで検証する概念実証として採用可能である。

特に次の点を評価する。

- 依存ライブラリなしで実行できる。
- 実際のHVACサンプルを使用している。
- 範囲と択一を区別している。
- 条件付き数量を意識している。
- 数量抽出と意味対応付けの境界を明記している。
- 既知の限界を文書化している。

一方、負数、境界の包含・非包含、複合的な上下限制約、原文保持、条件の誤伝播などが未対応である。したがって、現時点の判定は次のとおりとする。

> **概念実証として採用可能。本体ツールへの統合は、必須修正と失敗系テストを追加した後に再判定する。**

## 2. 必須修正

### 2.1 符号付き数値

現行の数値パターンは正数だけを対象としており、次のような環境仕様を抽出できない。

- `-10～50℃`
- `－20℃以上`
- `−5 °C`

ASCIIハイフン、全角マイナス、Unicodeマイナスを正規化し、符号付き数値として保持する。

### 2.2 上下限制約を一つの区間へ統合

`0℃以上50℃以下`を二つの独立した単一値ではなく、一つの区間として表現する。

```json
{
  "kind": "interval",
  "lower": { "value": 0, "inclusive": true },
  "upper": { "value": 50, "inclusive": true }
}
```

### 2.3 境界の包含・非包含

次を区別して保持する。

- 以上・以下: 境界を含む
- 超える・未満: 境界を含まない

単純な`min`/`max`だけでは、後続の充足判定を正確に実行できない。

### 2.4 原文と正規化文字列の分離

現在の`raw_text`は正規化後の文字列から生成されるため、厳密には原文ではない。

次を別々に保存する。

```json
{
  "source_text": "５０℃～６０℃",
  "normalized_text": "50℃~60℃"
}
```

正規化後の位置から元文字列へ戻れるよう、文字位置の対応も保持する。

### 2.5 比較結果を暫定判定として明示

`coverageGap()`は、同じ設計特性・同じ条件であることを確認していない。比較結果には次を明示する。

```json
{
  "provisional": true,
  "comparable": true,
  "assumptions": [
    "同じ設計特性として選択済み",
    "同じ運転条件",
    "単位換算不要",
    "抽出結果に要確認事項なし"
  ]
}
```

数量抽出だけを根拠に正式な適合・不適合を確定しない。

### 2.6 条件の誤伝播防止

現行実装は、条件候補を直後の数量へ付与する。このため、次のような文章で誤対応する可能性がある。

```text
50℃で停止し、電源電圧は220Vとする。
```

少なくとも次を検討する。

- 同一文・同一節内に限定する。
- 句点やセミコロンを越えない。
- 条件値と対象値の最大距離を設定する。
- 動詞・接続表現を確認する。
- 自動確定せず、確信度付き`condition_candidates`として保持する。

## 3. 推奨修正

### 3.1 単位比較はcanonical値で行う

`0℃から50°C`は意味的に同一単位だが、`fromToRe`は生文字列を比較するため取りこぼす。

`unitInfo(...).canonical`で比較し、異なる場合も黙って破棄せず`needs_review`付きの候補として残す。

### 3.2 スラッシュ表現の意味を確定しない

`50/60 Hz`の`/`は、両対応、択一、別機種、比率など複数の意味を持ち得る。

抽出段階では構文だけを保持する。

```json
{
  "kind": "alternatives",
  "options": [50, 60],
  "selection_semantics": "unknown",
  "needs_review": "スラッシュの意味は文脈判定が必要"
}
```

### 3.3 周辺条件を後段へ引き渡す

`三相AC 220 V`から`220 V`だけを抽出すると、電源比較に必要な条件を失う。

数量オブジェクトへ、`AC`、`DC`、`単相`、`三相`、`定格`などの周辺語を保持する領域を設ける。

```json
{
  "quantity": 220,
  "unit": "V",
  "context_tokens": ["定格電源", "三相", "AC"]
}
```

### 3.4 逆転範囲を検出する

`50～0℃`のように下限が上限を超える場合、原文を自動修正せず、`needs_review`を付与する。

## 4. 推奨スキーマ

単一値、片側制約、範囲を区間モデルへ統一する。

```json
{
  "source_text": "0℃以上50℃未満",
  "normalized_text": "0℃以上50℃未満",
  "quantity": {
    "kind": "interval",
    "lower": {
      "value": 0,
      "inclusive": true
    },
    "upper": {
      "value": 50,
      "inclusive": false
    }
  },
  "unit": {
    "source": "℃",
    "canonical": "degC",
    "dimension": "temperature"
  },
  "context": {
    "property": null,
    "subject": null,
    "state": null,
    "tokens": []
  },
  "extraction": {
    "confidence": 0.98,
    "warnings": []
  }
}
```

`50℃以下`のような片側条件は、反対側を`null`として表現する。

## 7. 設計原則

> 数量抽出は、数値・単位・演算子・範囲を構造化する工程であり、その値が何の設計特性を表すかを確定する工程ではない。

> 数量比較は、同じ設計特性・対象・状態・条件であることが確認されるまでは暫定結果として扱う。

> 正規化によって原文を失わず、すべての抽出・変換・比較結果から出典へ戻れる状態を維持する。
