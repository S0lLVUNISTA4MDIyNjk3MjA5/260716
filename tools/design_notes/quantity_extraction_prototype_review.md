## 1. 総合評価

本プロトタイプは、数量抽出を実データで検証する概念実証として採用可能である。

特に次の点を評価する。

- 依存ライブラリなしで実行できる。
- 実際のHVACサンプルを使用している。
- 範囲と択一を区別している。
- 条件付き数量を意識している。
- 数量抽出と意味対応付けの境界を明記している。
- 既知の限界を文書化している。

一方、負数、境界の包含・非包含、複合的な上下限制約、原文保持、条件の誤伝播などが未対応である。したがって、現時点の判定は次のとおりとする。

> **概念実証として採用可能。本体ツールへの統合は、必須修正と失敗系テストを追加した後に再判定する。**

## 2. 必須修正

### 2.1 符号付き数値

現行の数値パターンは正数だけを対象としており、次のような環境仕様を抽出できない。

- `-10～50℃`
- `－20℃以上`
- `−5 °C`

ASCIIハイフン、全角マイナス、Unicodeマイナスを正規化し、符号付き数値として保持する。

### 2.2 上下限制約を一つの区間へ統合

`0℃以上50℃以下`を二つの独立した単一値ではなく、一つの区間として表現する。

```json
{
  "kind": "interval",
  "lower": { "value": 0, "inclusive": true },
  "upper": { "value": 50, "inclusive": true }
}
```

### 2.3 境界の包含・非包含

次を区別して保持する。

- 以上・以下: 境界を含む
- 超える・未満: 境界を含まない

単純な`min`/`max`だけでは、後続の充足判定を正確に実行できない。

### 2.4 原文と正規化文字列の分離

現在の`raw_text`は正規化後の文字列から生成されるため、厳密には原文ではない。

次を別々に保存する。

```json
{
  "source_text": "５０℃～６０℃",
  "normalized_text": "50℃~60℃"
}
```

正規化後の位置から元文字列へ戻れるよう、文字位置の対応も保持する。

### 2.5 比較結果を暫定判定として明示

`coverageGap()`は、同じ設計特性・同じ条件であることを確認していない。比較結果には次を明示する。

```json
{
  "provisional": true,
  "comparable": true,
  "assumptions": [
    "同じ設計特性として選択済み",
    "同じ運転条件",
    "単位換算不要",
    "抽出結果に要確認事項なし"
  ]
}
```

数量抽出だけを根拠に正式な適合・不適合を確定しない。

### 2.6 条件の誤伝播防止

現行実装は、条件候補を直後の数量へ付与する。このため、次のような文章で誤対応する可能性がある。

```text
50℃で停止し、電源電圧は220Vとする。
```

少なくとも次を検討する。

- 同一文・同一節内に限定する。
- 句点やセミコロンを越えない。
- 条件値と対象値の最大距離を設定する。
- 動詞・接続表現を確認する。
- 自動確定せず、確信度付き`condition_candidates`として保持する。

## 3. 推奨修正

### 3.1 単位比較はcanonical値で行う

`0℃から50°C`は意味的に同一単位だが、`fromToRe`は生文字列を比較するため取りこぼす。

`unitInfo(...).canonical`で比較し、異なる場合も黙って破棄せず`needs_review`付きの候補として残す。

### 3.2 スラッシュ表現の意味を確定しない

`50/60 Hz`の`/`は、両対応、択一、別機種、比率など複数の意味を持ち得る。

抽出段階では構文だけを保持する。

```json
{
  "kind": "alternatives",
  "options": [50, 60],
  "selection_semantics": "unknown",
  "needs_review": "スラッシュの意味は文脈判定が必要"
}
```

### 3.3 周辺条件を後段へ引き渡す

`三相AC 220 V`から`220 V`だけを抽出すると、電源比較に必要な条件を失う。

数量オブジェクトへ、`AC`、`DC`、`単相`、`三相`、`定格`などの周辺語を保持する領域を設ける。

```json
{
  "quantity": 220,
  "unit": "V",
  "context_tokens": ["定格電源", "三相", "AC"]
}
```

### 3.4 逆転範囲を検出する

`50～0℃`のように下限が上限を超える場合、原文を自動修正せず、`needs_review`を付与する。

## 4. 推奨スキーマ

単一値、片側制約、範囲を区間モデルへ統一する。

```json
{
  "source_text": "0℃以上50℃未満",
  "normalized_text": "0℃以上50℃未満",
  "quantity": {
    "kind": "interval",
    "lower": {
      "value": 0,
      "inclusive": true
    },
    "upper": {
      "value": 50,
      "inclusive": false
    }
  },
  "unit": {
    "source": "℃",
    "canonical": "degC",
    "dimension": "temperature"
  },
  "context": {
    "property": null,
    "subject": null,
    "state": null,
    "tokens": []
  },
  "extraction": {
    "confidence": 0.98,
    "warnings": []
  }
}
```

`50℃以下`のような片側条件は、反対側を`null`として表現する。

## 7. 設計原則

> 数量抽出は、数値・単位・演算子・範囲を構造化する工程であり、その値が何の設計特性を表すかを確定する工程ではない。

> 数量比較は、同じ設計特性・対象・状態・条件であることが確認されるまでは暫定結果として扱う。

> 正規化によって原文を失わず、すべての抽出・変換・比較結果から出典へ戻れる状態を維持する。
