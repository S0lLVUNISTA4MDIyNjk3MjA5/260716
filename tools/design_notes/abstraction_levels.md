## 1. 背景：なぜこの議論をしているか

現行の3ツール（PDF→JSON、Excel→JSON、JSON A/Bトレース照合）は、`usage_guide.md`に記載の通り「様式プロファイルとタグ辞書を人間が事前に用意する」ことを前提にしている。この前提を緩和し、次の操作へ近づけられないかを検討している。

1. PDFとExcelを読み込む
2. ツールが構造・列役割・共通概念を自動認識する
3. 関連付けと条件比較を自動実行する
4. 人間は「要確認」の項目だけ判断する

辞書ファイルや様式プロファイルは、人間が事前作成するのではなく、ツール内部で自動生成・更新する。必要な場合のみ、上級者向けに編集・入出力機能を提供する（ファイル形式自体は廃止しない）。

この検討の中心的な問いは、**自動化を進める際、どこまでをAI（生成AI／LLM）に依存させ、どこまでを非AIのツールロジック（ルール・統計処理・構造抽出）で代替できるか**である。過度なAI依存を避けたいという問題意識に基づく。

## 2. 中心となる設計原則

改訂の結果、本プロジェクトは「抽象化ツール」ではなく、**出典を失わない意味対応付け・条件比較エンジン**として捉えるのが適切という結論に至った。核となる原則は次の3つ。

> **原則1（処理の分離）** 本ツールは、関連候補の発見・意味対応付け・要求値と設計値の抽出・適合性評価を**独立した処理**として実行し、それぞれの結果・確信度・根拠を保持する。

> **原則2（タグの位置づけ）** タグは候補抽出の補助情報であり、設計特性の同一性や要求充足を確定する情報ではない。タグの一致だけを根拠として、意味対応または要求充足を確定してはならない。

> **原則3（判断の分離）** 「関連している」「標準仕様で満たす」「顧客対応後に満たす」を別々に判断する。1つの信頼度や1つの「対応あり」状態へ統合しない。

## 3. 「タグ」と「意味対応付け」を分離する

初版の最大の誤りは、「語彙的集約（タグ化）」と「意味対応付け」を同一工程に含めていたことである。両者は別物として分離する。

- **タグ**：似た話題をまとめ、照合候補を絞り込むための補助情報。多対多でよく、確定情報ではない。
- **意味対応付け**：PDFとExcelの2つの記述が、同じ設計特性を扱っているかを判断する独立した処理。

例として、次はすべて「温度」タグに分類できるが、相互に数値比較できるとは限らない。

```text
周囲温度 ↔ 使用温度範囲    同じ設計特性の可能性が高い
周囲温度 ↔ 保管温度        異なる設計特性
周囲温度 ↔ 異常停止温度    異なる設計特性
```

この区別は、現行ツールの検証で実証された事実と噛み合う。`samples/hvac_trace_sample_small/verification_report.md`の検証で、「2.1 使用環境」の要求文が「運転騒音」（Excel側）と信頼度0.80でベクトル類似マッチした事例があった。根拠は「運転」という一般語の偶然の共通のみで、意味的関連はなかった。これは**タグ・文字列類似だけでは意味対応を確定できない**ことの実例であり、原則2の裏付けになっている。

### 意味対応付けで比較する情報

意味対応付けでは、単語の類似だけでなく、少なくとも次を比較する。

- **対象**：製品、部品、冷媒、周辺環境など
- **特性**：温度、電圧、能力、騒音など
- **状態**：通常運転、停止、保管、試験、異常時など
- **位置**：室内、室外、吸込、吹出など
- **条件**：連続、一時的、定格、最大、最小など
- **数量種別と単位**
- **否定語・除外語**
- **出典文脈**：章・節・Excel分類など

共通設計特性は、タグとは別のフィールドとして保持する。

```json
{
  "tags": ["温度", "環境条件"],
  "design_property": {
    "concept_id": "environment.ambient_operating_temperature",
    "label": "周囲使用温度",
    "subject": "製品",
    "state": "通常運転",
    "quantity_type": "temperature"
  }
}
```

## 4. 判断処理を分離する（原則3の詳細）

4章の温度の例には、少なくとも3種類の異なる判断が含まれる。

1. PDF要求とExcelの使用温度行は関連している
2. 標準機種の上限40℃では、要求上限50℃を満たさない
3. 顧客対応後の上限50℃では要求を満たす

これらを1つの信頼度や1つの「対応あり」へまとめず、独立した結果として保持する。

| 判断 | 結果例 |
|---|---|
| 関連性 | 同じ設計特性として関連あり |
| 標準適合性 | 10℃不足 |
| 対応後適合性 | 充足 |
| 総合状態 | 顧客対応により充足 |

## 5. 確信度も分離する

判断を分離するのに対応して、確信度も最低3種類を独立して管理する。

- **抽出確信度**：数値・単位・条件を正しく取得できたか
- **関連確信度**：PDF要求とExcel行が同じ設計特性か
- **充足判定確信度**：設計値が要求を満たすと判断できるか

関連確信度が高くても、「一時的」「約50℃」「特定条件下のみ」などの記述があれば、充足判定確信度は低くなり得る。この分離により、現行の確信度階層（高＝自動採用／中＝採用するが要確認／低＝候補提示／判定不能＝既定値＋理由表示）を、判断の種類ごとに独立して適用できる。

## 6. 処理工程モデル（6工程）

初版の5段階モデルに「意味対応付け」を独立工程として加え、6工程へ再整理する。

| 工程 | 内容 | 非AI（ツールロジック）だけで実現できるか |
|---|---|---|
| 1. 構造抽出 | 章・節・表・Excelの行列を認識する | **可能・現行ツールで実装済み** |
| 2. 概念抽出 | 温度・電源・騒音・能力などの設計特性候補を抽出する | **部分的に可能**。既存語彙からの抽出に限定すれば非AI |
| 3. 意味対応付け | 「周囲温度」と「使用温度範囲」などを共通設計特性へ対応付ける | **候補生成は可能／確定は不可**（下記の重要な留保を参照） |
| 4. 条件・数量構造化 | 数値・単位・範囲・以上/以下・運転条件などを構造化する | **可能・現行ツール未実装** |
| 5. 比較・横断分析 | 充足判定・差分・未対応・外れ値・網羅性を計算する | **可能**（4の構造化データが前提） |
| 6. 説明・推論 | 背景・理由・影響などを言語化する | **AI依存** |

LLMは全面的に使わず、非AI処理で判断できない意味対応（工程3の確定）や説明（工程6）に限定して使用する。

### 工程3・4に関する重要な留保：「抽出」と「意味解釈」を分ける

工程4の数量抽出（`0～50℃`という値・単位を取り出す）は非AIで実現できる。しかし、その値が「周囲使用温度／保管温度／試験温度／異常停止温度」のどれを指すかを**確定**するには、周辺文脈と工程3の意味対応付けが必要になり、非AIだけでは確信度が低い。

したがって数量データは、値と単位だけでなく次を併せ持つ構造とする。

- 設計特性（工程3の対応付け結果）
- 対象
- 状態（運転・保管・試験など）
- 比較演算子・範囲
- 出典
- 確信度

## 7. トレーサビリティ：可逆性ではなく検証可能性

初版は工程①〜④を「処理として可逆」と表現したが、これは不正確だった。クラスタリングや統計的集約は、結果だけを保存すると元情報を失う。そのため「処理が可逆」ではなく、**すべての判断に来歴を残して元データへ戻れる「検証可能性」**を保証する、と改める。

保存すべき情報：

- 元ファイル、ページ、章、節
- Excelシート、行、セル
- 元文章・元セル値
- 正規化前後の値
- 適用したルール
- 対応付けの根拠
- 使用した辞書・モデルのバージョン
- 各段階の確信度
- 人間による修正履歴

これは現行ツールが既に持つ出典保持（PDF側の`source_page`/`source_path`、Excel側の`source_record`/`source_row`）の延長線上にあり、意味対応付け・確信度・修正履歴を追加する形で拡張できる。

## 8. 実装優先順位

人間による辞書・様式の事前準備を減らすという目的を優先する場合、次の順序を提案する（レビュー担当AIの提案を採用）。

1. Excelの範囲・ヘッダー・列役割の自動認識
2. 概念抽出と共通設計特性への意味対応付け
3. PDF・Excel共通辞書の自動生成・再利用
4. 数値・単位・条件の構造化
5. 標準仕様と顧客対応後の充足判定
6. 統計的な傾向・外れ値検出
7. 低確信度項目に限定したLLM支援

### 優先順位2に関する留保（本文書で残す論点）

レビューは「数量的構造化（4）だけを先に実装しても、辞書・様式を人間が準備する問題は残るため、意味対応付け（2）までの自動化を優先したい」とする。この目的設定には同意する。

ただし、意味対応付けは6工程の中で**最も非AI自動化が難しく、確信度が低くなりやすい**工程であり、「過度にAIに依存しない」という本プロジェクトの原則と最も強く緊張する。緊張を解くため、優先順位2の「自動化」を次のように読み分けることを提案する。

- 意味対応付けの**候補生成と確信度算出**は自動化する（非AIヒューリスティック＋ブラウザ内埋め込みモデルで、対象・特性・状態を突き合わせ、関連確信度を出す）。
- 意味対応付けの**確定**は自動化しない。関連確信度が高＝自動採用、中＝採用するが要確認、低＝候補提示にとどめ人間が確定、判定不能＝既定値＋理由表示、とする。

この読み分けは、5章の「関連確信度を独立管理する」という提案とそのまま接続する。すなわち優先順位2は「意味対応付けの確定を自動化する」ではなく「意味対応付けの**候補生成と関連確信度の算出**を自動化する」と解釈する。

## 9. 初版（5段階モデル）からの主な差分

| 論点 | 初版（5段階） | 本版（6工程） |
|---|---|---|
| 意味対応付け | 段階②「語彙的集約」に混在 | 工程3として独立分離 |
| タグの扱い | 集約結果の一種 | 候補絞り込みの補助情報と明確化（原則2） |
| 判断の粒度 | 段階③④で数値比較・統計 | 関連性/標準適合性/対応後適合性を分離（原則3・4章） |
| 確信度 | 単一の「確信度」を想定 | 抽出/関連/充足判定の3分離（5章） |
| トレーサビリティ | 「①〜④は可逆」 | 「可逆ではなく検証可能性（来歴保存）」（7章） |
| 数量抽象化 | 段階③に一括 | 抽出（工程4・非AI）と意味解釈（工程3・確定はAI/人間）を分離 |

## 10. 未決定事項・レビュー観点

引き続き検討・レビューしてほしい点。

1. **意味対応付けの候補生成を、非AI（ヒューリスティック＋ブラウザ内埋め込み）でどこまで実現できるか。** 対象・特性・状態・位置・条件の突き合わせを、辞書とルールでどこまで機械化でき、どこからLLM橋渡しが必要になるか。8章の「候補生成/確定の切り分け」が実装上も成立するかは未検証。
2. **`concept_id`（共通設計特性の識別子）の体系をどう設計するか。** 3章の例では`environment.ambient_operating_temperature`のような階層IDを想定したが、この語彙体系を自動生成するのか、案件横断の固定辞書を持つのか、両者のハイブリッドか。自動生成する場合、誤った概念統合（別特性を同一視する）のリスク管理が必要。
3. **工程5（統計的横断分析）に必要な最小データ量。** 少数レコード（今回のサンプルは5件）で外れ値・傾向を計算すると、誤ったパターンを検出するリスクがある。
4. **単位・条件表記の揺れ（全角/半角、範囲表現、複合単位、「約」「以上」「一時的」等の条件語）を、工程4の構造化ロジックでどこまで吸収できるか。**
5. **来歴保存（7章）のデータ量とパフォーマンス。** 全判断に完全な来歴を持たせると、ブラウザ完結・localStorage/IndexedDB前提の現行アーキテクチャで容量・速度が問題にならないか。

いずれも未着手・未実装であり、実装の是非を含めて議論中である。
