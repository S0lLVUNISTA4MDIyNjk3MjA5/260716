V12.1.15内蔵ヘルプ
このツールの目的
任意のJSON A/Bを読み込み、顧客要求・JSON A・JSON B/BOM・購買仕様書などの対応関係を自動候補化し、レビュー結果を保存するトレース照合ツールです。 自動判定はレビュー支援であり、最終判断はトレースマトリクス上で確認してください。

クイックスタート

  1. 照合プロファイルを選びます。
  2. JSON A に上流・照合元、JSON B に下流・比較対象を指定します。
  3. 読み込み・照合 を押します。任意JSONでは、読み込み後に照合キーを確認してから再照合します。
  4. 照合ロジック設定 でキーを確認し、必要なら 再照合して反映 を押します。
  5. トレースマトリクス で対応あり・要確認・A未対応・B未参照を確認します。

最初に設定する画面

最初は 照合ロジック設定 でJSON A/Bの照合キーを確認し、再照合してから トレースマトリクス で結果を確認してください。

V11.14 性能設定

長文帳票では、exact最大スコア到達時の早期確定、キーワード上限、あいまい候補上限で照合負荷を抑えます。元フィールド全文とコードは優先保持されます。設定変更後は再照合してください。

レビューの残し方

トレースマトリクスでレビュー判定とコメントを入力し、レビューJSON保存 を押します。次回はレビューJSON読込または差分ベースライン読込で継続できます。

V11.14.1 一括レビュー編集

標準トレースマトリクスでフィルタを適用し、判定を選んで対象全件へ一括適用できます。既存レビュー保護には「レビュー未入力の行だけ」を使います。

V12.0.1 入力アノテーション・タグ照合

JSON A/Bの各行へタグを付与し、供給源別に確認・追加・除外できます。Stage 3を明示的にONにすると、低頻度タグを候補剪定とDice照合へ使用します。

1. 基本操作手順
  1. 組み込みの汎用JSON（JSON A ↔ JSON B）プロファイルを使用します。
  2. ファイル名プレフィックスを必要に応じて入力します。Excel / SysML / JSON / SVGの保存ファイル名に使われます。
  3. JSON Aには上流側、要求側、照合元を入れます。例：顧客要求仕様、JSON A。
  4. JSON Bには下流側、実体側、比較対象を入れます。例：JSON A、JSON B/BOM、購買仕様書。
  5. 読み込み・照合を押します。
  6. トレースマトリクスで分類を確認します。
  7. 必要に応じて照合ロジック設定でキー・しきい値・業務辞書を調整し、再照合します。
  8. レビュー後、ExcelダウンロードまたはレビューJSON保存で結果を残します。
2. 照合プロファイル
プロファイル使う場面主な確認観点
汎用JSON（JSON A ↔ JSON B）任意構造のJSON同士を比較自動推定されたキー、タグ、照合方式の確認
事前設定は汎用JSONのみです。必要な場合はユーザー定義プロファイルをJSONから読み込めます。
3. 照合方式と判定の意味
方式意味扱い
exact正規化後の完全一致原則「対応あり」
code本文中コード、コード、略称コードの一致原則「対応あり」
tag統制語彙タグ共有（Dice係数）要確認
synonym手動辞書・業務辞書による同義語一致内容により要確認
auto-synonymJSONから自動生成した同義語候補要確認
hier親子ゲートにより降格された候補要確認。元方式と関連項目は根拠列に表示
partial包含・部分一致要確認。親子項目の誤照合に注意
fuzzybigram類似度による表記ゆれ一致要確認
vectorTF-IDF/特徴語ベースの文脈類似要確認
vector/fuzzy/auto-synonymは、正解の可能性がある候補を拾うための方式です。自動確定ではなく、人間レビュー前提で扱ってください。
4. トレースマトリクスの見方
分類意味レビュー観点
対応ありexact/codeなどで強く対応根拠が業務上妥当か確認
要確認類似・部分一致・文脈一致誤照合、親子項目、関連項目を確認
A未対応A側に対応先BがないA側漏れ、B側漏れ、購買仕様なしを確認
B未参照B側がA側から参照されていない根拠不明項目、未参照項目、未参照仕様を確認

同一B項目が複数Aから参照される場合は、B集約ビューでJSON B単位の被参照状況を確認してください。

5. トレースマトリクスのフィルタ・並べ替え

トレースマトリクス上部のフィルタ・並べ替えで、全列検索、分類、方式、レビュー状態、差分、信頼度範囲を指定できます。列ヘッダをクリックすると、その列で昇順/降順を切り替えます。

  • フィルタと並べ替えは画面表示だけに適用されます。
  • 通常のExcelダウンロードは監査用途での誤認を避けるため、常に全件を出力します。
  • 概要モードやB集約ビューでも、存在する列に対してフィルタ・並べ替えできます。
  • 一括レビュー編集は標準（行×行）表示だけで利用できます。フィルタ条件に一致した全件が対象で、初期500行表示の範囲だけには限定されません。
  • 「レビュー未入力の行だけ」をONにすると、既に判定済みの行を保護できます。直前の一括変更は同じ照合結果の間だけ元に戻せます。
手動関係・付け替え

機械が候補に出さなかったA–B関係を手動追加できます。付け替えは旧関係を誤対応として残し、新関係を手動追加する二つの操作として記録します。

  • 手動関係は詳細表・概要・グラフ・RO-Crateメタデータ・Excelなどのレビュー反映結果へ即時反映します。
  • 削除は履歴を残す論理削除です。機械候補や過去の監査情報を破壊しません。
  • JSON差し替えでAまたはBのIDが消えた判断は孤立レビューとして保持し、別行へ自動適用しません。
  • レビューJSON v2はレビュー、手動関係、付け替え履歴をまとめて保存・復元します。
  • 対応あり・誤対応・手動追加・付け替えは、学習利用がONの場合に教師ラベルへ同期されます。
Stage 1回帰修正
全文と分割句が同じフィールドは、exact専用のfullへ置き換えず従来のsegment/codeとして評価します。キーワード生成はV11.13の集合を再利用し、生成元分類・優先順位・上限だけを追加します。
レビュー・学習統合

人が確定した判断のうち、学習に適したものだけを既存のロジスティック回帰へ教師ラベルとして同期します。確定トレースへの反映と学習利用は分離されているため、学習できない場合でも人の判断は失われません。

  • 対応あり・手動追加・付け替え後の新関係は正例1、誤対応・付け替え前の旧関係は負例0です。
  • 部分対応・要確認・対象外は既定で教師対象外です。
  • トレースマトリクスの「学習利用」で同期のON/OFFを変更できます。
  • 理由コードは辞書・階層・範囲・データ品質・MLのどこを改善すべきかを分析するために保持します。
  • 手動追加ペアは機械候補に存在しなくても特徴量を強制計算します。計算不能時は「保留」として表示し、確定関係には影響しません。
  • 学習モデルの予測と人判断が異なる行だけを学習タブで抽出できます。再学習後も人の確定判断が優先されます。
人手レビュー反映済みトレース

レビュー判定は注釈ではなく、機械照合を人が確定・修正する判断として扱います。「対応あり」「部分対応」「要確認」は有効関係として残り、「誤対応」「対象外」はレビュー反映結果の有効件数・詳細表・グラフ・関係出力から除外されます。

  • レビュー反映結果: 人の判断を優先した確定結果です。
  • 機械照合結果: 人が修正する前の自動結果を監査できます。
  • 機械・人手差分: 機械判定と人判断が異なるペアだけを確認します。
  • 「未対応」はA項目に有効なB関係がない場合に自動算出され、A–Bペアのレビュー選択肢には使用しません。
  • レビュー変更では機械照合キャッシュを破棄せず、有効関係・詳細表・概要・グラフ・出力だけを更新します。
6. 入力アノテーション（Stage 2）

読み込み後、各行へ次の供給源から内部タグを生成します。タグは通常のJSON・Excel・ReqIF・SysML・RO-Crate出力には含まれません。

  • 明示列: tags / タグ / labels / hashtags
  • 業務辞書: synonymMapの基本語。本文に基本語または同義語が出た場合に付与
  • コード: 既存コード形式に加え、P/N表記などを抽出
  • 手動: 確認表で追加・除外。辞書再生成後も同一セッション内では保持
「タグを照合に使用」をOFFにすると、タグは確認・整備専用となり、V11.16.1相当の従来照合へ戻ります。タグを含む入力データが必要な場合は「アノテーション保存」を使用してください。
同じ明示タグを全行へ付けると、将来のタグ剪定効果が失われます。業務領域を区別できる具体的なタグを使用してください。
6. Stage 3 タグ照合エンジン

低頻度の有効タグをB側転置索引へ登録し、code候補、tag候補、fuzzy候補の順で決定的に評価します。非コードタグの信頼度は 0.88 × Dice係数 です。

  • 共有コードタグは既存のcode方式（既定0.95）へ昇格します。
  • B側で高頻度の辞書タグは表示には残りますが、候補剪定・Dice・fuzzy省略には使いません。
  • 非コードタグは1件共有でも、Diceが設定閾値以上ならfuzzy照合を省略できます。共有件数の下限はありません。
  • 共有タグが1件以上ありDiceが設定値以上の場合、exact/codeの確認後にvector・fuzzy・partialを省略します。
  • タグなし行、または共有可能な低頻度タグがない行は従来候補生成へ完全フォールバックします。
タグ方式は既定OFFの候補機能です。辞書・コードタグは有効キーペアの対象フィールドだけから生成し、自動生成同義語は辞書タグに使用しません。人が確定したレビュー・手動追加・付け替えは、再照合後も機械結果より優先されます。
6. JSONコード表示と照合処理の中止

入力JSON表タブの「JSONコード表示（成形付き）」で、JSON A/Bの原文JSONまたは正規化後の行データをインデント付き・色分け付きで確認できます。表示内容はコピーまたはJSON保存できます。

読み込み・再照合中は、入力欄下に処理インジケーターを表示します。処理が重い場合は照合中止を押すと、現在の処理単位が終わり次第停止します。中止後は前回結果が残る場合があります。

6. 業務辞書インポート

専門語・略語・日英表記ゆれは、業務辞書で補強できます。照合ロジック設定タブの業務辞書読込を使います。

JSON形式例：
{ "synonymMap": { "リチウムイオン電池": ["Li-ion Battery", "Li-ionバッテリ"], "回生ブレーキ": ["Regenerative Braking"] } }
CSV形式例：
base,synonym,category
リチウムイオン電池,Li-ion Battery,Battery
回生ブレーキ,Regenerative Braking,Braking
6. 表示粒度と概要モード

トレースマトリクスでは、標準（行×行）、概要（グループ集計）、B集約を切り替えられます。概要モードは照合を再実行せず、標準粒度の照合結果を親グループ単位にロールアップします。

  • 章節型JSONは source_section_id を親グループとして扱います。
  • 部品親子構造は parent_part_code などの親参照フィールドを検出し、直近親でグルーピングします。
  • 親キーが行として存在しない場合も、仮想親としてグループを成立させます。
  • 親でもあり子でもある部品は、親参照に従って直近親グループへ所属します。自分のグループ側には子の集約コンテナとして現れます。
  • 概要行は読み取り専用です。レビュー判定・コメントは概要行からドリルダウンした標準行で入力してください。
7. 照合ゲート（親子ゲート・属性ゲート）

親子ゲートは、項目階層上の親子にまたがる包含一致を hier 方式へ降格し、対応ありではなく要確認として残します。例として、子項目名の一部だけで親項目が exact 扱いになった場合、親側を要確認にします。

  • 階層ゲートはプロファイルの useHierarchyGate でON/OFFできます。既定はONです。
  • 属性ゲートはプロファイルJSONの gatingFields にカテゴリ・材質・系統などのキー候補を記載した場合だけ動作します。
  • 属性値が両側にあり不一致の場合、信頼度に mismatchFactor を掛け、exact/codeでも要確認にします。しきい値未満になった候補は通常どおり消えます。
  • 設定は「プロファイルJSON保存 → JSON編集 → プロファイルJSON読込」の流れで行います。例: {"gatingFields":[{"a":["category"],"b":["category"],"mismatchFactor":0.75}]}
7. レビュー保存と差分比較
  1. トレースマトリクスでレビュー判定・コメントを入力します。
  2. レビューJSON保存で現在のレビューと照合スナップショットを保存します。
  3. 次回、同じレビューを継続する場合はレビューJSON読込を使います。
  4. 前回結果との差分を見たい場合は差分ベースライン読込を使います。

差分は「変化なし / 新規 / 変更 / 消滅」で表示されます。

7. L3文チャンクと自動クラスタリング

L3文チャンクは、JSON A側の長文を文単位に分割して照合し、根拠列へ「該当文」を表示する機能です。既定はOFFです。照合ロジック設定でONにし、再照合してください。レビュー入力は従来どおり標準粒度のA_ID×B_IDで行います。

自動クラスタリングは、章節や親子構造が少ないフラットなJSONに対し、概要表示用の合成グループ（CLU）をメモリ上に作る機能です。照合スコアや分類には影響しません。再読み込みで消えます。

大規模データではクラスタリングをWorkerまたは分割実行で処理します。5000行超はブラウザ内実行を拒否し、外部前処理を推奨します。
8. レスポンス改善（Phase 4）
  • 任意JSONプロファイルでは、初期ロード時に照合を自動実行せず、まずキー確認を優先します。
  • 照合は小バッチで非同期実行し、進捗表示と中止操作を受け付けます。中止時は中途半端な結果を反映しません。
  • グラフ、照合結果一覧、モデルテキスト、JSONコード表示はタブを開いた時に遅延生成します。
  • 大きな表は初期500行表示とし、「さらに表示」で追加描画します。Excel出力は引き続き全件対象です。
8. 大規模データ利用時の注意

V11.5以降は候補生成型照合を使い、A×Bの全件総当たりを避けます。ただし、数千件以上ではブラウザのメモリやPC性能に依存します。

  • まず100件程度でキー設定を確認してください。
  • コード列・カテゴリ列がある場合は照合キーに含めてください。
  • vector/fuzzyのしきい値を下げすぎると候補が増えて重くなります。
  • 不要な文書行・提出書類行が混じる場合は、JSON側で除外または正規化してください。
