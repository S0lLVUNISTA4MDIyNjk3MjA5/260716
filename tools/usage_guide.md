# PDF仕様書とExcel設計項目のトレース照合ツール群

## 利用方法・対応範囲・運用仕様

**文書版:** 1.1
**対象ツール:**

- 仕様書PDF→JSON変換・確認ツール v1.18
- Excel→JSON変換・確認ツール v2.0.8
- JSON A/Bトレース照合ツール v12.1.15
- JSON A/Bトレース照合ツール簡易版 v1.5

---

## 1. 目的

本ツール群は、客先仕様書に記載された要求と、Excelの設計検討表に記載された設計項目との関連を確認するために使用する。

想定する業務では、人間が次の作業を行っている。

1. PDF形式の客先仕様書を読む。
2. 仕様書の各章・節・文章から、設計上確認すべき要求を把握する。
3. Excelにマトリクス形式で整理された設計項目を確認する。
4. 仕様書の要求とExcelの設計項目との対応関係を判断する。

本ツール群は、この作業を完全に自動決定するものではない。PDFとExcelを照合可能なJSONへ変換し、関連性が高い組み合わせを候補として提示することで、人間による確認作業を支援する。

最終的な対応関係、要求への適合性、設計上の妥当性は、人間が原文と設計情報を確認して確定する。

## 2. 想定するExcel帳票

### 2.1 基本構造

対象Excelは、次の条件を満たすマトリクス形式を想定する。

- ヘッダーは1行で構成される。
- 1行が1つの設計項目を表す。
- 行方向には設計項目が並ぶ。
- 列方向には設計項目の属性、標準機種情報、顧客対応、検討結果などが並ぶ。

例を以下に示す。

| 設計項目 | 標準機種 | 顧客対応 | 検討内容 | 検討結果 |
|---|---|---|---|---|
| 冷房能力 | 10 kW | 能力増加 | 客先要求値との比較 | 12 kWで設計 |
| 電源電圧 | AC 200 V | 電源仕様変更 | 使用機器の対応確認 | AC 220 V対応 |
| 使用温度 | 0～40 ℃ | 高温環境対応 | 部品定格の確認 | 0～50 ℃対応 |

この場合、照合の基本単位はExcelの各セルではなく、行全体である。

たとえば「冷房能力」の行は、次の情報を組み合わせた1件の設計項目として扱う。

- 設計項目ラベル：冷房能力
- 標準機種：10 kW
- 顧客対応：能力増加
- 検討内容：客先要求値との比較
- 検討結果：12 kWで設計

> **技術的な補足**：Excel→JSON変換ツールは、プロファイルの`text_columns`に指定した列の値を`' / '`区切りで連結し、1本の照合用テキスト（`trace_text`）を生成する。行に含まれる元の列値そのものは行データ（`source_record`）として失われずに残るが、**自動照合が直接利用するのは連結後の1本のテキストであり、「どの列に由来する一致か」は自動では区別されない**。この前提は3章・7章・10章で改めて扱う。

### 2.2 現行ツールに適した条件

現行ツール群は、次の条件で特に利用しやすい。

- 1行に複数の独立した設計項目を混在させない。
- 設計項目名を示す列が明確である。
- 標準機種と顧客対応の情報が別の列に整理されている（連結後も人間が読んで区別できる程度に、列の内容が簡潔であることが望ましい）。
- 見出し行、注記行、小計行などを設計項目行と区別できる。
- 照合結果として「どの設計項目行に関連するか」が分かればよい。列単位の切り分けまでは自動照合に求めない。

## 3. 照合の基本単位

### 3.1 PDF側

PDF側では、仕様書を次の単位に分解する。

- 文書
- 章
- 節
- 本文・段落
- 箇条書き項目
- 表の行

各文章には、可能な範囲で次の出典情報を保持する。

- 元PDFファイル名
- 章番号・章タイトル
- 節番号・節タイトル
- ページ番号
- PDF上の位置情報
- 構造化JSON内の出典パス
- 元の文章

> **技術的な補足**：仕様書PDF→JSONツールは、本文を段落（block）単位、箇条書きを項目（item）単位、表を行（row）単位で個別の照合レコードへ分解する（照合用JSON出力オプション`text_unit`/`list_unit`/`table_unit`）。各レコードは所属する章番号・節番号を保持するため、「PDFのどの章の文章か」という粒度は標準機能で十分に満たされる。

### 3.2 Excel側

Excel側では、原則として1行を1件の設計項目レコードへ変換する。

各レコードには、次の情報を含めることができる。

- 元Excelファイル名
- シート名
- 行番号
- 設計項目ラベル
- 標準機種情報
- 顧客対応情報
- 検討内容
- 検討結果
- 備考
- 照合タグ
- 確認状態

これらの列の値は、変換後も行データとして丸ごと保持される。ただし照合本文として使われるのは、プロファイルで指定した複数列を連結した1本のテキストである。列を分けて照合したい場合は、4.2節・9.1節の運用方法に従う。

### 3.3 照合関係

照合ツールは、PDF側の文章レコードとExcel側の設計項目行レコードを比較する。

基本的な対応関係は次のとおりである。

```text
PDFの章・節・文章
        ↓
関連性の候補を抽出
        ↓
Excelの設計項目行
        ↓
人間が原文と設計内容を確認
        ↓
確定・要確認・除外などを判断
```

1件のPDF文章が複数のExcel行に関連する場合や、複数のPDF文章が1件のExcel行に関連する場合もある。そのため、対応関係は1対1に限定しない。

## 4. 各ツールの役割

### 4.1 仕様書PDF→JSON変換・確認ツール

#### 役割

PDF仕様書を、後段の照合ツールで扱える構造化JSONへ変換する。

#### 主な作業

1. 客先仕様書PDFを読み込む。
2. 章・節・本文・箇条書き・表の抽出結果を確認する。
3. 誤分割、文字化け、OCR誤認識などを修正する。
4. 照合対象外の文章を除外する。
5. 要求内容に照合タグを付ける。
6. 確認済みJSONまたは照合用JSONを出力する。

#### 確認すべき点

- 章番号と節番号が正しく認識されているか。
- 1つの要求文章が不自然に分割されていないか。
- 複数の要求が1つの文章へ過度に結合されていないか。
- 数値、単位、温度、圧力、許容差などが正しく抽出されているか。
- 表のヘッダーと行の対応が正しいか。
- 出典ページと元の文章を追跡できるか。

### 4.2 Excel→JSON変換・確認ツール

#### 役割

Excelの設計検討表を、1行1設計項目のJSONへ変換する。

#### 主な作業

1. Excelファイルを読み込む。
2. 対象シートを選択する。
3. 変換対象範囲を指定する。
4. ヘッダー行とデータ開始行を指定する。
5. 列名とJSONフィールドの対応を確認する。
6. 空白セル、結合セル、データ型などの変換状態を確認する。
7. どの列を照合本文（`text_columns`）に含めるかを決め、各設計項目行に照合タグを付ける。
8. 確認済みJSONまたは照合用JSONを出力する。

#### 推奨する列の役割

| Excel列 | 推奨する用途 |
|---|---|
| 設計項目 | 行の代表名称、照合結果の表示名 |
| 分類・機能 | 設計項目のグループ化 |
| 標準機種 | 基準となる既存仕様 |
| 顧客対応 | 客先要求によって変更・確認する内容 |
| 検討内容 | 設計上確認した事項 |
| 検討結果 | 設計値、採否、対応内容 |
| 根拠・備考 | 計算、規格、判断理由など |

#### 確認すべき点

- ヘッダー行が正しく指定されているか。
- 設計項目以外のタイトル行や小計行が混入していないか。
- 設計項目列が行の代表名称として使用されているか。
- 標準機種と顧客対応の情報が区別されているか。
- `text_columns`に指定した複数列が1本のテキストへ連結される仕様を理解した上で、列の組み合わせを決めているか。
- 空白や結合セルの補完によって誤った値が入っていないか。
- 数値と単位が失われていないか。
- PDF側と同じ意味のタグが付けられているか。

### 4.3 JSON A/Bトレース照合ツール

#### 役割

PDF側JSONをJSON A、Excel側JSONをJSON Bとして読み込み、関連する文章と設計項目行の候補を提示する。

#### 主な作業

1. JSON AにPDF由来の照合用JSONを指定する。
2. JSON BにExcel由来の照合用JSONを指定する。
3. 照合キーの組み合わせを確認する。
4. 読み込み・照合を実行する。
5. 照合結果一覧で関連候補を確認する。
6. 必要に応じてトレースマトリクスとナレッジグラフを確認する。
7. 候補となったExcel行を開き、標準機種・顧客対応・検討内容・検討結果のどの記載が根拠なのかを人間が確認する。
8. 人間が結果を確認し、確定、要確認、除外などを判断する。
9. 照合結果をExcelなどへ出力する。

#### 照合に利用する情報

- PDFの章・節タイトル
- PDF本文
- Excelの設計項目名
- Excelの照合本文（標準機種情報・顧客対応情報・検討内容・検討結果などを連結した1本のテキスト）
- PDFとExcelに付与された照合タグ
- 一致した語句
- 文字列類似度
- コード、数値、単位などの一致

> 標準機種情報と顧客対応情報は、照合エンジンの中では区別されずに1本のテキストとして扱われる。**一致語句がどちらの列に由来するかは自動では判定されない**ため、候補が提示された場合は必ずExcel行を開いて該当箇所を確認する（7章参照）。

### 4.4 簡易版照合ツール

簡易版は、詳細な設定やレビュー機能を減らし、照合結果一覧とナレッジグラフを短時間で確認する用途に適する。フル版と同じ自動照合エンジンを使用するが、照合ロジックの詳細設定・トレースマトリクス・学習機能・Excel出力・手動レビュー編集などは持たない。

次の用途で利用しやすい。

- 操作デモ
- 初回の照合結果確認
- タグや入力データの品質確認
- フル版を使用する前の一次評価

正式なレビュー、手動関係編集、トレースマトリクス、詳細出力が必要な場合はフル版を使用する。

## 5. 推奨する業務フロー

### フェーズ1：事前準備

1. 対象PDFと対象Excelを決める。
2. Excelが1行1設計項目になっていることを確認する。
3. Excelのどの列を照合本文に含めるか（`text_columns`の構成）を決める。
4. PDFとExcelで共通使用するタグ辞書を準備する。
5. 対象外とする文章・行の基準を決める。

### フェーズ2：PDFの構造化

1. PDFを変換ツールへ読み込む。
2. 章・節・文章の抽出状態を確認する。
3. 文章の修正、結合、分割、除外を行う。
4. 照合タグを付ける。
5. 未確認項目や未登録タグを解消する。
6. JSON Aを出力する。

### フェーズ3：Excelの構造化

1. Excelを変換ツールへ読み込む。
2. 対象シート、ヘッダー行、データ開始行、範囲を指定する。
3. 1行1設計項目として正しく変換されたことを確認する。
4. 設計項目、標準機種、顧客対応、検討結果などの列を確認する。
5. 照合タグを付ける。
6. 未確認項目や未登録タグを解消する。
7. JSON Bを出力する。

### フェーズ4：自動照合

1. 照合ツールへJSON AとJSON Bを読み込む。
2. 自動選択された照合キーを確認する。
3. ID、ハッシュ値、内部管理番号が意味照合へ使用されていないことを確認する。
4. 必要に応じて照合キーを修正する。
5. 照合を実行する。

### フェーズ5：人間による確認

1. 照合結果一覧をPDF文章基準で確認する。
2. 関連候補となったExcel行を確認する。
3. 候補行を開き、標準機種・顧客対応のどちらの記載が一致根拠なのかを人間が読み分ける。
4. PDF原文の条件とExcelの設計内容を比較する。
5. 関連しているか、情報が不足しているか、誤照合かを判断する。
6. 必要に応じて手動で関係を追加、修正、除外する。
7. コメントや判断理由を記録する。

### フェーズ6：結果出力

1. 未確認結果が残っていないか確認する。
2. 信頼度が低い結果や1対多の結果を重点確認する。
3. 照合結果一覧、トレースマトリクス、ナレッジグラフの内容が一致することを確認する。
4. 必要な形式で結果を出力する。

## 6. 照合タグの位置付け

照合タグは、PDF文章とExcel行の候補を効率的に絞り込むために使用する。

例：

- 冷房能力
- 暖房能力
- 使用温度
- 電源電圧
- 消費電力
- 圧力
- 騒音
- 質量
- 寸法
- 耐振動
- 保守性

### 6.1 タグ運用の原則

- PDFとExcelで同じタグ辞書を使用する。
- 同義語は代表タグへ統一する。
- 文書ごとに表記を変えない。
- タグを細分化しすぎない。
- タグだけで最終判定しない。
- 数値条件は本文と設計値を直接確認する。

たとえば「冷却能力」「冷房出力」「クーリング能力」が同じ意味で使用される場合、代表タグを「冷房能力」とし、他の表記は同義語として辞書に登録する。

## 7. 人間が確認すべき内容

照合ツールが提示する信頼度は、関連性の候補を並べるための指標であり、要求への適合を保証するものではない。

人間は少なくとも次の内容を確認する。

### 7.1 関連性

- PDF文章とExcel設計項目が同じ機能・部位を扱っているか。
- 一致した語句が偶然の一致ではないか。
- 共通タグが実際の内容を正しく表しているか。
- **一致語句が標準機種情報・顧客対応情報・検討内容・検討結果のどの記載に由来するか**、Excel行を開いて確認する（照合結果一覧の表示だけでは列は区別されない）。

### 7.2 数値と単位

- 要求値と設計値の単位が一致しているか。
- 単位換算が必要ではないか。
- 最小値、最大値、範囲、許容差の意味が一致しているか。
- 「以上」「以下」「未満」「超える」などの条件を正しく比較しているか。

### 7.3 成立条件

- 温度条件が一致しているか。
- 圧力条件が一致しているか。
- 電源条件や運転モードが一致しているか。
- 定常、起動時、異常時などの状態が一致しているか。
- 顧客固有条件が標準機種情報と混同されていないか（連結された照合本文の中で読み違えやすい点のため、特に注意する）。

### 7.4 網羅性

- PDF要求に対応するExcel行が存在するか。
- Excel側に検討漏れがないか。
- 1つの要求が複数の設計項目へ展開される場合、必要な行がすべて対応しているか。

## 8. 現行ツールで対応できる範囲

### 8.1 対応できる内容

- PDFを章・節・文章（段落・箇条書き・表行）へ構造化する。
- PDFの文章、箇条書き、表行を照合対象にする。
- PDFの出典ページや構造上の位置を保持する。
- Excelを1行1設計項目としてJSON化する。
- Excelの指定した複数列の値をまとめて1つの照合本文として利用する（列ごとの重み付けや区別は行わない）。
- PDFとExcelへ共通タグを付与する。
- タグ、一致語句、文字列類似度などから関連候補を抽出する。
- 1対1、1対多、多対1の候補を表示する。
- 照合結果一覧、トレースマトリクス、ナレッジグラフを表示する。
- 人間によるレビューと結果出力を行う。

### 8.2 現行仕様の前提

- Excelの基本照合単位はセルではなく行である。
- Excelのヘッダーは基本的に1行である。
- Excelの1行が1つの設計項目を表す。
- 行内の複数列は1本の照合本文へ連結され、自動照合は列を区別しない。列の区別が必要な確認は人間がExcel行を開いて行う。
- 最終判定は人間が行う。
- 照合結果は関連性の候補であり、要求への自動適合判定ではない。

### 8.3 現時点で優先しない拡張

今回の想定用途では、次の拡張は必須ではない。

- Excelの全セルを個別の照合レコードへ変換する機能
- PDF文章とExcelセル番地の厳密な1対1対応
- 複数段ヘッダーの完全な階層モデル化
- 自動照合だけで適合・不適合を確定する機能

これらは、将来セル単位の監査証跡や自動適合判定が必要になった場合の拡張候補とする。実現する場合、Excel側の変換方式を「1行1設計項目」から「1行1セル（設計項目×列ラベル）」の縦持ち形式へ変更し、列ラベルを独立した列として持たせる前処理が必要になる。

## 9. 運用上の注意点

### 9.1 Excelの行構成

- 途中のタイトル行、小計行、空行は変換対象から除外する。
- 1行内に複数の独立した設計項目を記載しない。
- 設計項目名だけで内容が分からない場合は、分類や機能列を併用する。
- 標準機種情報と顧客対応情報を同じセルへ混在させない。
- `text_columns`に含める列数を増やしすぎない。連結後のテキストが長くなるほど、一致語句がどの列由来か人間が読み分けにくくなる。

### 9.2 照合キー

- 設計項目、要求本文、顧客対応、検討結果、タグなど、意味のある項目を使用する。
- 自動生成ID、ハッシュ値、行管理番号を意味照合の中心にしない。
- コード照合を使用する場合は、PDFとExcelで同じ体系の業務コードであることを確認する。

### 9.3 信頼度

- 高信頼度でも原文確認を省略しない。
- 低信頼度でも重要な要求は除外せず確認する。
- 信頼度のしきい値だけで要求の有無を判断しない。
- タグが不足すると関連候補が見つからない場合がある。

### 9.4 ナレッジグラフ

- 大量データでは対象要求や信頼度で絞り込む。
- グラフは照合結果を視覚化するものであり、独自の照合結果を生成しない。
- 照合結果一覧とグラフで、同じ表示名、タグ、関係、信頼度を使用する。

---

## 10. 推奨する照合結果一覧の確認手順

現行の照合結果一覧・トレースマトリクスは、Excel側を「設計項目名」「照合本文（連結済み）」「タグ」などの行単位の項目で表示する。標準機種・顧客対応・検討結果を個別の列としては表示しない。そのため、次の手順で確認する。

| 区分 | 表示項目 |
|---|---|
| PDF側 | 文書名、章・節、文章、ページ、タグ |
| Excel側 | ファイル名、シート名、設計項目、照合本文（連結済み）、タグ |
| 照合情報 | 信頼度、一致タグ、一致語句、比較フィールド、照合根拠 |
| 人間の判断 | 確認状態、採否、コメント、確認者、確認日時 |

結果一覧では、最初に次の3点が分かることを重視する。

1. PDFのどの要求文章か。
2. Excelのどの設計項目行か。
3. なぜ関連候補になったか。

その上で、候補行が示された時点で「入力JSON表」や元Excelを開き、標準機種・顧客対応・検討内容・検討結果のどの記載が根拠かを人間が確認する。これは自動化されないため、レビュー手順として明文化しておく。

## 12. まとめ

現行ツール群は、次の条件を満たす業務であれば、PDF仕様書とExcel設計項目のトレース照合に概ね対応できる。

- Excelのヘッダーが1行である。
- Excelの1行が1つの設計項目を表す。
- PDFのどの文章がExcelのどの設計項目行に関連するかを確認したい。
- 最終判断は人間が行う。
- 標準機種・顧客対応など列ごとの違いは、自動照合の絞り込みには使わず、人間が候補行を開いて確認する運用で対応できる。

この用途では、Excelをセル単位に細分化する大幅な仕様変更よりも、次の整備を優先することが有効である。

1. PDFとExcelで共通のタグ辞書を使用する。
2. PDF文章とExcel設計項目行の粒度を適切に管理する。
3. 意味のある照合キーを選択する。
4. 一致タグ、一致語句、比較フィールドなどの照合根拠を明示する。
5. 人間の確認結果を保存し、一覧・マトリクス・グラフ・出力へ一貫して反映する。
6. 「一致語句がどの列に由来するかは自動判定されない」という前提を、レビュー担当者全員に周知する。

したがって、本ツール群は「自動適合判定システム」ではなく、**PDF要求とExcel設計項目行の関連候補を提示し、人間によるトレーサビリティ確認を支援するツール群**として位置付けるのが適切である。

## 13. 技術補足（プロファイル設計者向け参考情報）

本章は、Excelプロファイル（様式JSON）を設計する担当者向けの技術参考情報である。運用担当者は読み飛ばしてよい。

| 項目 | 内容 |
|---|---|
| PDF側の照合用JSON形式 | `chapter-section-trace-v1`。分解粒度は`text_unit:"block"`（段落）、`list_unit:"item"`（箇条書き）、`table_unit:"row"`（表の行）。各レコードは`chapter_number`/`section_number`を保持する。 |
| Excel側の照合用JSON形式 | `excel-row-trace-v1`。1行＝1レコード。 |
| 照合本文の生成方法 | プロファイル`output.text_columns`に指定した列の非空値を`' / '`で連結して`trace_text`を生成する（`textFromColumns`関数）。 |
| 元データの保持 | Excel側の各レコードには、連結前の行データ全体が`source_record`として格納されるが、照合エンジンおよび標準の照合結果一覧・トレースマトリクスはこれを参照しない。 |
| 列を区別して照合したい場合 | Excel側を「1行1設計項目」ではなく「1行1セル（設計項目×列ラベル）」の縦持ち形式へ事前に変換し、列ラベルを`category_column`などへ割り当てる。ツール自体にマトリクスの自動展開（unpivot）機能はない。 |
